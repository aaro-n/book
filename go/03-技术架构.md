## 🔄 数据流设计

### 1. 文档上传流程

```
用户选择PDF文件
    ↓
前端验证（大小、格式）
    ↓
上传到后端
    ↓
后端验证文件完整性
    ↓
保存到S3存储
    ↓
计算SHA256哈希（去重）
    ↓
提取文本并分词
    ↓
┌────────────────────┐
│ 存储处理结果              │
├──────────────────────┤
│ 1. PG - document_content表      │
│    - raw_text: 原文本           │
│    - segmented_text: 分词结果   │
│    - word_frequency: 词频统计   │
│                            │
│ 2. 搜索引擎（异步）             │
│    - ES或Meilisearch        │
│    - 自动处理分词和索引         │
└──────────────────────┘
    ↓
更新文档元数据
    ↓
返回用户：上传成功
```

### 2. 用户标记流程

```
用户在阅读器中操作
    ↓
┌─────────────────┐
│ 选择操作类型           │
├────────────────────────┤
│ 高亮文本 → highlights表       │
│ 添加注释 → annotations表      │
│ 书签页面 → bookmarks表        │
└──────────────┘
    ↓
分词用户输入
    ↓
存储到PG
    ↓
异步索引到搜索引擎
    ↓
返回用户：操作成功
    ↓
源PDF文件保持不变（关键！）
```

### 3. 搜索流程

```
用户输入搜索词
    ↓
应用层分词
    ↓
┌─────────────────────┐
│ 查询搜索引擎（优先级）          │
├─────────────────────┤
│ 1. 尝试主搜索引擎（ES默认）    │
│    ├─ 命中 ✓ → 返回结果        │
│    └─ 失败 → 继续              │
│                 │
│ 2. 尝试备选搜索引擎     │
│    ├─ 命中 ✓ → 返回结果        │
│    └─ 失败 → 继续              │
│                    │
│ 3. 降级到PG搜索（最后手段）   │
│    └─ ILIKE查询（简单但可用）  │
└───────────────────────┘
    ↓
高亮匹配词
    ↓
返回结果给用户
```

## 💾 内存管理策略

### 启动时内存分布（512MB限制）

```
基础框架              ~20MB
├─ Gin框架          5MB
├─ AWS SDK        8MB
└─ 其他库       7MB

分词引擎          ~60MB
├─ GSE词库           20MB
├─ 自定义词库         5MB
└─ 缓存            35MB

搜索适配           ~80MB（如启用）
├─ ES客户端          20MB
├─ 搜索缓存          60MB

业务数据              ~80MB
├─ 连接池         20MB
├─ 缓存池            50MB
└─ 临时数据          10MB

运行时开销            ~100MB
├─ Go堆              60MB
├─ Goroutine栈       30MB
└─ 缓冲区            10MB

═══════════════════
总计               ~340MB（有250MB余量）
```

### 内存优化技巧

| 优化项 | 效果 | 实现方式 |
|-----|------|---|
| **仅用GSE** | -40MB | 不加载完整Jieba词库 |
| **LRU缓存** | -20MB | 限制缓存条数（1000）和大小 |
| **对象池** | -15MB | 重用[]byte对象 |
| **延迟加载** | -10MB | 需要时加载搜索引擎 |
| **自动GC** | - | 400MB阈值触发强制GC |

## 🔐 安全考虑

### 认证和授权

1. **用户认证** - JWT token
   ```go
   // 流程：Login → 生成Token → 存入Redis → 后续请求验证
   // Token过期：1小时
   // Refresh Token：7天
   ```

2. **权限验证** - 用户隔离
   ```go
   // 所有操作都验证user_id
   // 确保用户只能访问自己的文档和标记
   // 数据库约束：UNIQUE(user_id, document_id)
   ```

### 数据保护

1. **源文件不可修改**
   ```sql
   -- PDF文件完全由用户标记表控制
   -- 源PDF永远存储在S3，不动
   -- 标记表有CREATE/UPDATE/DELETE权限
   ```

2. **密码安全**
   ```go
   // 使用bcrypt哈希，不存储明文
   // 密码验证比对时间恒定（防时序攻击）
   ```

3. **传输安全**
   ```
   -- HTTPS传输
   -- CORS配置严格
   -- 敏感操作需要CSRF token
   ```

## 📊 性能指标

### 目标指标

| 指标 | 目标 | 说明 |
|------|---|
| **API延迟** | <200ms | 99%请求 |
| **分词速度** | <100ms | 1000词以内 |
| **搜索响应** | <300ms | 包含排序和分页 |
| **文件上传** | <5s | 100MB文件 |
| **内存占用** | <512MB | Go应用专用 |
| **并发连接** | >100 | 同时在线用户 |

### 性能优化手段

1. **分词引擎选择**
   - 默认GSE（快速轻量）
   - 必要时启用Jieba（精准）
   - 结果缓存避免重复分词

2. **数据库优化**
   - 索引优化（GIN索引for tsvector）
   - 连接池管理
   - 慢查询监控

3. **搜索优化**
   - 搜索结果缓存
   - 热词预加载
   - 索引合并（定期）

4. **缓存策略**
   - Redis缓存热点数据
   - LRU缓存本地热词
   - 缓存穿透保护

```

├─ API请求数/秒

├─ 响应时间分布

├─ 慢查询日志

├─ 搜索延迟
├─ 索引大小
├─ 查询成功率
└─ 回源请求率
```

### 日志策略

```go
// 日志级别
DEBUG   - 开发调试信息（本地开发）
INFO    - 关键业务事件（上传、搜索）
WARN    - 警告但不影响（降级、重试）
ERROR   - 错误但可恢复（连接失败）
FATAL   - 致命错误（启动失败）

// 日志内容
- 请求ID（trace_id）追踪整个请求链路
- 用户ID隔离用户数据
- 操作耗时（milliseconds）
- 错误堆栈（仅FATAL/ERROR）
```
