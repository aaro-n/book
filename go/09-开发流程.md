# 开发流程 - AI编程的标准工作流

## 🎯 核心原则

**一切为了方便AI编程**：
- 分解任务为最小单位
- 每次只编写一个小文件
- 明确的输入输出规范
- 完整的错误处理

---

## 📋 标准工作流程

### 第0步：需求分析和规划

**您的工作**：

```markdown
# 需求：实现用户注册功能

## 功能描述
用户提交email、password，系统创建用户账户

## 输入和输出
- 输入：email(string), password(string)
- 输出：user_id(int), 或错误

## 业务规则
1. email必须唯一
2. password最少8字符，包含字母和数字
3. password存储为bcrypt哈希
4. 返回user_id，用于后续操作

## 受影响的模块
1. models: 新增/修改User结构体
2. repository: 新增CreateUser方法
3. service: 新增Register方法
4. handler: 新增RegisterHandler
5. tests: 添加单元测试

## 涉及表
- users表：新增记录

## 边界情况
- email已存在 → 返回错误
- password太弱 → 返回错误
- 数据库错误 → 返回500错误
```

### 第1步：数据模型定义

**给AI的提示词**：

```markdown
# 任务：定义用户数据模型

## 需求
定义User结构体用于注册功能

## 位置
internal/models/user.go

## 字段
- ID: int
- Email: string (唯一)
- Password: 不要存这个字段（存hash）
- PasswordHash: string (bcrypt hash，min 60字符)
- DisplayName: string (可选)
- CreatedAt: time.Time
- UpdatedAt: time.Time

## 约束
- 不超过50行
- 每字段都要JSON tag

## 示例
```go
type User struct {
    ID         int       `json:"id"`
    Email        string    `json:"email"`
    PasswordHash string    `json:"-"`  // 不暴露到JSON
    DisplayName  string    `json:"display_name"`
    CreatedAt    time.Time `json:"created_at"`
    UpdatedAt    time.Time `json:"updated_at"`
}
```

## 常量定义
- MinPasswordLength = 8
- MaxDisplayNameLength = 100
```

**输出验证**：
- ✓ 字段齐全
- ✓ 有JSON tag
- ✓ 行数<50
- ✓ 密码Hash字段有json:"-"标签

### 第2步：数据访问层

**给AI的提示词**：

```markdown
# 任务：实现用户仓库的CreateUser方法

## 需求
在users表中创建新用户记录

## 位置
internal/repository/user_repo.go

## 方法签名
func (r *UserRepository) CreateUser(ctx context.Context, user *models.User) error

## 逻辑
1. 验证email不重复（先检查）
2. INSERT新用户到users表
3. 返回错误或nil

## SQL参考
```sql
INSERT INTO users (email, password_hash, display_name, created_at, updated_at)
VALUES ($1, $2, $3, NOW(), NOW())
```

## 错误处理
- 如果email重复 → 返回ErrEmailExists
- 如果数据库错误 → 返回ErrDatabaseError

## 约束
- 不超过25行
- 使用context用于超时控制

## 注意
- 不验证password强度（service层的职责）
- 不计算hash（调用者已做）
- 仅做数据库操作
```

**输出验证**：
- ✓ 方法签名正确
- ✓ 有context参数
- ✓ 错误处理完整
- ✓ 行数<30

### 第3步：业务逻辑层

**给AI的提示词**：

```markdown
# 任务：实现注册服务

## 需求
处理用户注册的完整业务流程

## 位置
internal/service/auth_service.go

## 方法
func (s *AuthService) Register(ctx context.Context, email, password string) (int, error)

## 业务流程
1. 验证email格式（如xxx@xxx.com）
2. 验证password强度（最少8字符，包含字母和数字）
3. 使用bcrypt加密password
4. 调用repository.CreateUser
5. 返回user_id或错误

## 密码验证规则
- 最少8字符
- 必须包含至少1个字母
- 必须包含至少1个数字

## Bcrypt使用
```go
import "golang.org/x/crypto/bcrypt"

hash, err := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)
```

## 错误处理
- 无效email格式 → 返回ErrInvalidEmail
- 密码过弱 → 返回ErrWeakPassword
- email已存在 → 返回ErrEmailExists
- 数据库错误 → 返回ErrDatabaseError

## 约束
- 不超过40行
- 所有error都要处理
- 返回值：(user_id, error)

## 测试
给出至少2个测试用例：
1. 正常注册
2. 密码过弱
```

**输出验证**：
- ✓ 业务逻辑清晰
- ✓ 密码验证完整
- ✓ 错误处理全面
- ✓ 行数<50

### 第4步：HTTP处理层

**给AI的提示词**：

```markdown
# 任务：实现注册API处理器

## 需求
处理POST请求 /api/auth/register

## 位置
internal/handler/auth_handler.go

## 方法
func (h *AuthHandler) Register(c *gin.Context)

## 请求格式
```json
{
  "email": "user@example.com",
  "password": "SecurePass123"
}
```

## 响应格式
成功(200)：
```json
{
  "success": true,
  "data": {
    "user_id": 123
  }
}
```

失败(400/500)：
```json
{
  "success": false,
  "error": {
    "code": "WEAK_PASSWORD",
    "message": "密码强度不足"
  }
}
```

## 处理流程
1. 绑定JSON请求
2. 验证请求格式
3. 调用service.Register
4. 处理错误
5. 返回响应

## 错误映射
- ErrInvalidEmail → 400 + INVALID_EMAIL
- ErrWeakPassword → 400 + WEAK_PASSWORD
- ErrEmailExists → 400 + EMAIL_EXISTS
- ErrDatabaseError → 500 + INTERNAL_ERROR

## 约束
- 不超过30行
- 不涉及业务逻辑（调用service）
- 仅处理HTTP细节

## 使用util提供的响应函数
util.SuccessResponse(c, 200, data)
util.ErrorResponse(c, code, appError)
```

**输出验证**：
- ✓ 请求绑定正确
- ✓ 错误处理完整
- ✓ 响应格式统一
- ✓ 行数<40

### 第5步：单元测试

**给AI的提示词**：

```markdown
# 任务：编写注册功能的单元测试

## 位置
internal/service/auth_service_test.go

## 测试用例
1. TestRegisterSuccess - 正常注册成功
2. TestRegisterWeakPassword - 密码过弱
3. TestRegisterInvalidEmail - Email格式错误
4. TestRegisterEmailExists - Email已存在

## 测试框架
使用testing包 + testify/assert

## 示例结构
```go
func TestRegisterSuccess(t *testing.T) {
    // Arrange（准备）
    repo := NewMockUserRepository()
    service := NewAuthService(repo)
    
    // Act（执行）
    userID, err := service.Register(context.Background(), "test@example.com", "SecurePass123")
    
    // Assert（验证）
    assert.NoError(t, err)
    assert.Greater(t, userID, 0)
}
```

## Mock对象
创建MockUserRepository，模拟数据库操作
- 正常返回
- 返回email已存在错误

## 约束
- 不超过80行（所有测试）
- 至少4个测试用例
- 覆盖主要场景和错误情况
```

**输出验证**：
- ✓ 测试用例覆盖全面
- ✓ 使用了Mock对象
- ✓ 断言正确
- ✓ 没有真实数据库依赖

## 📝 代码审查清单

每个功能完成后，逐项检查：

### 代码规模
- [ ] 函数不超过30行
- [ ] 文件不超过200行
- [ ] 结构体不超过5个方法
- [ ] 参数不超过5个（多个则用结构体）

### 错误处理
- [ ] 所有error都被处理
- [ ] 有清晰的ErrorCode定义
- [ ] 没有silently ignore错误
- [ ] error信息清晰

### 命名和注释
- [ ] 变量名清晰易懂
- [ ] 函数名动宾式（如GetUser）
- [ ] Public函数有godoc注释
- [ ] 复杂逻辑有行内注释

### 测试
- [ ] 有单元测试
- [ ] 测试覆盖主要场景
- [ ] 测试覆盖错误情况
- [ ] Mock对象恰当

### 性能
- [ ] 没有明显的性能问题
- [ ] 没有N+1查询
- [ ] 没有死循环
- [ ] 大对象没有复制

### 并发安全
- [ ] 如果涉及并发，有正确的同步
- [ ] 没有race condition
- [ ] 没有deadlock

## 💡 常见问题处理

### Q1: 代码生成的不满意怎么办？

**A**: 提供更详细的提示词
```markdown
# 原始提示词（不够清楚）
实现获取文档的函数

# 改进的提示词（清晰）
# 任务：实现GetDocumentByID方法

## 位置
internal/repository/document_repo.go

## 方法签名
func (r *DocumentRepository) GetDocumentByID(ctx context.Context, docID, userID int) (*models.Document, error)

## 逻辑
1. 查询documents表
2. 验证document_id和user_id匹配（防止访问他人文档）
3. 排除is_deleted=true的记录
4. 返回Document指针或错误

## 错误处理
- 未找到 → sql.ErrNoRows → 返回ErrDocumentNotFound
- 用户无权访问 → 返回ErrUnauthorized

## 约束
- 不超过20行
- 必须包含user_id验证
```

### Q2: 一个文件太多代码，怎么拆分？

**A**: 按职责拆分成多个文件
```
不好：
user_service.go (200行)
├─ Register
├─ Login
├─ GetProfile
├─ UpdateProfile
└─ ChangePassword

改进：
auth_service.go (80行)
├─ Register
└─ Login

user_profile_service.go (70行)
├─ GetProfile
└─ UpdateProfile

user_password_service.go (60行)
└─ ChangePassword
```

### Q3: 错误太多，AI生成的代码不对怎么办？

**A**: 原因通常有三个：
1. **提示词不清楚** → 提供更多context和示例
2. **任务粒度太大** → 拆分成更小的步骤
3. **AI理解有误** → 说出问题在哪，让它重新生成

```
您的反馈：
"这个方法没有检查user_id，导致用户可以访问他人文档。
重新生成，必须在第2行添加用户权限检查。"
```

## 📚 推荐的工作模式

### 高效模式（推荐）

1. **集中讨论** - 先把完整需求和约束说清楚
2. **一次成型** - 利用提示词模板，一次生成好代码
3. **快速反馈** - 审查完立即反馈，而不是等所有代码都写完
4. **持续优化** - 代码上线后，根据反馈继续优化

### 容易失败的模式 ❌

```
❌ 模式1：特别详细的需求
您： "我需要一个注册功能...（50行需求）"
问题：AI容易过度设计，而不是最小可行版本

❌ 模式2：一次实现整个大功能
您： "实现整个用户认证模块"
问题：容易出错，难以调试

❌ 模式3：模糊的需求
您： "优化代码"
问题：AI不知道怎么优化，结果可能变差

✅ 模式4：清晰的小任务（推荐）
您： "实现UserRepository.GetByID方法...（约束和示例）"
问题：少，代码质量高
```

## ✅ 总结

**高效AI编程的关键**：
1. **任务分解** - 分成最小单位（<2h每个）
2. **明确约束** - 行数、错误处理、性能等
3. **提供示例** - 预期的输入输出格式
4. **快速反馈** - 立即指出问题和改进方向
5. **持续迭代** - 不求一步到位，逐步优化

**记住**：
- 小任务 = 更高的成功率
- 清晰的需求 = 更好的代码质量
- 快速反馈 = 更快的迭代
