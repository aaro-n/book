# Go 技术方案定位 — 与 Rust 方案完整对照

> **定位**：本文档阐述 Go 作为主选方案的理由，并提供与 Rust 方案的对比参考。
> **前提**：前端（React + TypeScript）、数据库（PostgreSQL 18）、基础设施（S3/Redis/搜索引擎）保持不变，仅后端语言选型不同。
> 
> Rust 版对照见 `../rust/10-Rust备选方案.md`。

**关键差异**：
- Go 用 `internal/` 包级隔离；Rust 用 `src/` 模块树 + `pub(crate)` 可见性控制

- Go 的 `go.mod` 声明模块路径；Rust 的 `Cargo.toml` 声明 crate 元信息

## 三、代码实现对照

### 3.1 应用入口

**Go：**
```go
func main() {
    cfg, err := config.Load()
    if err != nil {
        panic(err)
    }
    db, err := config.InitDB(cfg)
    if err != nil {
        panic(err)
    }
    defer db.Close()

    segmenter, _ := initSegmenter(cfg)
    repos := repository.NewRepositories(db)
    services := service.NewServices(repos, segmenter)
    handlers := handler.NewHandlers(services)

    r := gin.Default()
    setupRoutes(r, handlers)
    r.Run(fmt.Sprintf(":%d", cfg.Server.Port))
}
```

**Rust：**
```rust
#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let cfg = config::load()?;
    config::init_logger(&cfg);
    let pool = config::init_db(&cfg).await?;
    let segmenter = init_segmenter(&cfg)?;
    let state = AppState::new(pool, redis, s3, segmenter);
    let app = router::build(state);
    let listener = tokio::net::TcpListener::bind(&addr).await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

> Go 更简洁：goroutine 隐式，`defer` 清晰；Rust 显式声明 `#[tokio::main]` 异步运行时。

### 3.2 路由注册

**Go (Gin)：**
```go
func setupRoutes(r *gin.Engine, h *handler.Handlers) {
    auth := r.Group("/api/v1/auth")
    auth.POST("/register", h.Auth.Register)
    auth.POST("/login", h.Auth.Login)

    api := r.Group("/api/v1")
    api.Use(middleware.AuthRequired())
    api.POST("/documents/upload", h.Document.Upload)
}
```

**Rust (Axum)：**
```rust
pub fn build(state: AppState) -> Router {
    let public = Router::new()
        .route("/api/v1/auth/register", post(auth::register))
        .route("/api/v1/auth/login", post(auth::login));
    let protected = Router::new()
        .route("/api/v1/documents", post(document::upload))
        .layer(axum::middleware::from_fn_with_state(state.clone(), auth::require_auth));
    Router::new().merge(public).merge(protected).with_state(state)
}
```

> Gin 的 `Group()` 模式更直观；Axum 的类型安全路由在编译期捕捉更多错误。

### 3.3 Handler 实现

**Go：**
```go
func (h *AuthHandler) Register(c *gin.Context) {
    var req RegisterInput
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(400, gin.H{"error": err.Error()})
        return
    }
    user, err := h.service.Register(c.Request.Context(), &req)
    if err != nil {
        c.JSON(500, gin.H{"error": err.Error()})
        return
    }
    c.JSON(201, user)
}
```

**Rust：**
```rust
pub async fn register(
    State(state): State<AppState>,
    Json(input): Json<RegisterInput>,
) -> Result<(StatusCode, Json<UserResponse>), AppError> {
    input.validate()?;
    let user = state.auth_service.register(input).await?;
    Ok((StatusCode::CREATED, Json(user.into())))
}
```

> Go 更简洁直接；Rust 通过 `?` 运算符和 `IntoResponse` 实现更优雅的错误传播。

### 3.4 Repository 层

**Go：**
```go
func (r *UserRepo) Create(ctx context.Context, user *models.User) error {
    return r.db.QueryRowContext(ctx,
        `INSERT INTO users (email, username, password_hash)
         VALUES ($1, $2, $3) RETURNING id, created_at`,
        user.Email, user.Username, user.PasswordHash,
    ).Scan(&user.ID, &user.CreatedAt)
}
```

**Rust（sqlx 编译期校验 SQL）：**
```rust
pub async fn create(pool: &PgPool, user: &CreateUser) -> Result<User, sqlx::Error> {
    sqlx::query_as!(User,
        r#"INSERT INTO users (email, username, password_hash, display_name)
           VALUES ($1, $2, $3, $4)
           RETURNING id, email, username, display_name, created_at, updated_at"#,
        user.email, user.username, user.password_hash, user.display_name,
    ).fetch_one(pool).await
}
```

> `sqlx::query_as!` 在编译期连接数据库检查 SQL 语法和返回类型，字段不匹配直接编译报错。Go 的 SQL 错误只会在运行时发现。

## 四、内存占用对照

| 场景 | Go | Rust | 说明 |
|------|-----|------|------|
| 空载运行 | 15-25 MB | 5-8 MB | Rust 省 60%+（无 GC） |
| 加载 GSE/Jieba 分词词典 | +20-110 MB | +15-80 MB | Rust 省 ~25% |
| 处理单个 PDF 上传 (10MB) | +30-50 MB | +10-15 MB | GC 延迟释放导致 Go 峰值更高 |
| 稳态运行 (100 并发) | 80-200 MB | 30-60 MB | Rust 显著更省内存 |
| **512MB 限制下的余量** | 较紧张 | 非常充裕 | 需注意 Go GC 调优 |

**Go 内存优势**：虽然 Rust 内存更省，但 Go 的 GC 经过多年优化，1.22+ 版本在 512MB 环境下表现良好。配置 `GOGC=50` 可有效控制内存峰值。

### Goroutine vs Tokio Task

| | Goroutine | Tokio Task |
|---|-----------|------------|
| 初始栈大小 | 2-4 KB | ~64 bytes (无栈) |
| 10 万并发内存 | ~200-400 MB | ~6-8 MB |
| 编程心智负担 | ⭐ 极低 | ⭐⭐⭐ 中等 |

> 内存差异巨大，但实际场景 10 万并发极少见。正常 <1000 并发下，两者差异不显著。

## 五、中文分词对照

| 能力 | Go (GSE) | Rust (jieba-rs) | 推荐 |
|------|----------|-----------------|------|
| 基础分词 | ✅ | ✅ | 持平 |
| 自定义词典 | ✅ `AddToken()` | ✅ `add_word()` | 持平 |
| **热更新词典** | ✅ **GSE 原生支持** | ⚠️ 需自行实现 reload | **Go 胜** |
| 词性标注 | ✅ | ✅ | 持平 |
| 搜索引擎模式 | ✅ | ⚠️ 文档较少 | **Go 胜** |
| 内存占用 | 20-40 MB | 15-30 MB | Rust 略优 |
| AI 编程友好度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | **Go 胜** |

> **关键差异**：GSE 原生支持词典热更新，无需额外开发。jieba-rs 需要包装 `RwLock` 实现。

## 六、Docker 部署对照

### Go Dockerfile（多阶段构建）

```dockerfile
# ===== 构建阶段 =====
FROM golang:1.22-alpine AS builder

RUN apk add --no-cache git ca-certificates

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o pdf-manager ./cmd/main

# ===== 运行阶段 =====
FROM alpine:3.20
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /app/pdf-manager /usr/local/bin/
COPY config/ /etc/pdf-manager/

EXPOSE 8080
CMD ["pdf-manager"]
```

### 镜像大小对比

| | Go | Rust |
|---|-----|------|
| 构建镜像 | ~600 MB | ~1.2 GB（首次） |
| 运行镜像 (Alpine) | ~15-25 MB | ~10-18 MB |
| Scratch 镜像 | ~8 MB (CGO_ENABLED=0) | ~5 MB |
| **编译时间（冷）** | **~30 秒** ✅ | 2-5 分钟 |
| **增量编译** | **1-3 秒** ✅ | 10-30 秒 |

> Go 编译速度是 Rust 的 10-100 倍，对开发迭代体验影响巨大。

## 七、AI 编程效率对照 — 选择 Go 的核心理由

| 场景 | Go | Rust | 说明 |
|------|-----|------|------|
| CRUD 开发 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | Go 简单性让 AI 一次生成可用率很高 |
| 并发编程 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | Goroutine 心智负担极低 |
| 错误处理 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 显式 `if err != nil` 对 AI 最友好 |
| 序列化 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 双方都好 |
| PDF 处理 | ⭐⭐⭐⭐ | ⭐⭐⭐ | pdfcpu 更成熟，AI 训练数据更多 |
| 中文分词 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | GSE 生态丰富 |
| **总体 AI 可用率** | **~80%** | **~50%** | AI 生成 Go 代码一次可用的概率高 60% |

### AI 修复循环对比

```
Go 典型流程：
  提示词 → AI生成 → 编译通过 → 运行 → 可能有小bug → 修1-2轮 → 完成
  总耗时：~5 分钟

Rust 典型流程：
  提示词 → AI生成 → 编译失败 → AI修复 → 编译失败 → AI修复 → 编译通过
  → 运行 → 发现 .clone() 滥用 → AI重构 → 编译通过 → 完成
  总耗时：~15-20 分钟
```

> **结论**：Go 的 AI 返工轮次是 Rust 的 1/3，对于依赖 AI 编程的零经验团队，这是决定性因素。

## 八、综合分析

### 8.1 Go 明显优势的场景

| 场景 | 原因 |
|------|------|
| **零经验团队 + AI 编程** | 简单语法，AI 训练数据最多 |
| **快速迭代** | 编译秒级，修改即运行 |
| **CRUD 为主的业务系统** | Gin + GORM/lib/pq 生态成熟 |
| **中文分词** | GSE 热更新原生支持 |
| **PDF 处理** | pdfcpu 成熟稳定 |
| **团队协作** | 学习曲线平缓，1-2 天上手 |

### 8.2 Rust 明显优势的场景

| 场景 | 原因 |
|------|------|
| **内存严格受限 (<256MB)** | 无 GC，内存省 60%+ |
| **极致性能要求** | 零成本抽象，SIMD 优化 |
| **编译期安全要求极高** | 类型系统 + 借用检查 |
| **WebAssembly 嵌入** | wasm-bindgen 生态最佳 |

### 8.3 风险清单

| 风险 | 严重程度 | 缓解措施 |
|------|----------|----------|
| 512MB 内存紧张 | 🟡 中 | `GOGC=50` + `GOMEMLIMIT=400MiB` |
| GC 暂停 | 🟢 低 | Go 1.22 GC 暂停 <1ms |
| 并发竞态 | 🟡 中 | `go test -race` + channel 模式 |
| 类型安全不如 Rust | 🟢 低 | 充分测试 + lint (golangci-lint) |
