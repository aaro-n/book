# 测试策略 — AI 编程模式下的 Rust 测试规范

> **设计原则**：全程 AI 编程，测试也是 AI 写的。本文档教 AI 写正确的测试——给什么 prompt，怎么写断言，怎么 mock。
> **参考基准**：Kavita（xUnit + NSubstitute + SQLite In-Memory）、Komga（JUnit5 + MockK + SpringBootTest + ArchUnit）、Calibre-Web（pytest + Flask test client）。

---

## 一、三大项目测试方案调研

| | Kavita (C#) | Komga (Kotlin) | Calibre-Web (Python) |
|---|---|---|---|
| **测试框架** | xUnit + NSubstitute | JUnit5 + MockK + SpringMockK | pytest |
| **DB 测试** | SQLite `:memory:` + EF Core | SQLite 临时文件 + Flyway | 真实 DB 或 mock |
| **文件系统** | `MockFileSystem` | **Jimfs**（Google 内存文件系统） | 真实临时目录 |
| **HTTP 测试** | Flurl.Http | `MockMvc` + 自定义安全注解 | Flask test client |
| **架构测试** | 无 | **ArchUnit**（DDD 分层规则） | 无 |
| **覆盖率** | coverlet | JaCoCo | 无强制 |
| **测试文件数** | ~90+ | 93 | 少量 |
| **测试比例** | 95% 单元 + 5% 集成 | 40% 单元 + 60% 集成 | 手动测试为主 |

**关键发现**：
1. **三个人都用 SQLite 替代真实数据库做测试**——Kavita 用 `:memory:`，Komga 用临时文件 `random.uuid.sqlite`
2. **Komga 有最强的测试文化**：93 个测试文件 + ArchUnit 架构规则 + `@Nested` 分组 + Given-When-Then 风格
3. **Kavita 的 Builder 模式值得借鉴**——`LibraryBuilder`、`SeriesBuilder` 让测试数据构造可读
4. **Calibre-Web 测试最弱**——传统 SSR 应用，测试难写，不要学

## 二、Rust 测试体系总览

### 2.1 三层测试

```
┌─────────────────────────────────────────────┐
│ L1: 单元测试 (#[test])                       │
│ 位置: src/ 同文件底部 mod tests {}            │
│ 范围: 纯函数、数据结构、工具函数               │
│ 依赖: 无外部依赖，不连 DB/Redis/S3           │
│ 目标: 100+ 个，每个 <20ms                    │
├─────────────────────────────────────────────┤
│ L2: 服务层测试 (#[tokio::test])              │
│ 位置: src/ 同文件底部                         │
│ 范围: Service 层，mock 外部依赖               │
│ 依赖: mock 的 Repository / S3 / Redis        │
│ 目标: 50+ 个，每个 <100ms                    │
├─────────────────────────────────────────────┤
│ L3: API 集成测试 (tests/ 目录)               │
│ 位置: tests/ 独立文件                        │
│ 范围: 完整 HTTP 请求 → 响应                  │
│ 依赖: 真实 PG (SQLite 替代) + Mock S3/Redis   │
│ 目标: 30+ 个，每个 <500ms                    │
└─────────────────────────────────────────────┘
```

## 三、L1：单元测试

### 3.1 测什么

| 测 | 不测 |
|----|------|
| ✅ 纯函数（hash、mask、encode） | ❌ 框架代码（axum Router 注册） |
| ✅ 数据结构转换（From/Into） | ❌ 简单的 getter/setter |
| ✅ 校验逻辑（validate_password） | ❌ 外部 crate 的功能 |
| ✅ 工具函数（key 生成、URL 拼接） | ❌ SQL 查询语法（sqlx 编译检查已覆盖） |

## 四、L2：服务层测试（mock 依赖）

### 4.1 Mock 策略

```
Service 依赖：

  Repository → mock（不连真实 DB）
  S3 Client → mock（不连真实 S3）
  Redis     → mock（不连真实 Redis）

不 mock 的内容：
  jieba-rs（纯本地，不需要 mock）
  bcrypt（纯本地，不需要 mock）
```

### 4.2 用 mockall 定义 mock

```rust
// src/service/auth_service.rs

use mockall::automock;

/// Repository trait——让 mockall 自动生成 MockUserRepo
#[cfg_attr(test, automock)]
#[async_trait]
pub trait UserRepo: Send + Sync {
    async fn find_by_email(&self, email: &str) -> Result<Option<User>, AppError>;
    async fn create(&self, user: &NewUser) -> Result<User, AppError>;
    async fn update_password(&self, user_id: i32, hash: &str) -> Result<(), AppError>;
}

pub struct AuthService<R: UserRepo> {
    repo: R,
}

impl<R: UserRepo> AuthService<R> {
    pub async fn register(&self, input: RegisterInput) -> Result<User, AppError> {
        // 1. 检查邮箱唯一
        if self.repo.find_by_email(&input.email).await?.is_some() {
            return Err(AppError::Conflict("邮箱已注册".into()));
        }
        // 2. 哈希密码
        let hash = hash_password(&input.password)?;
        // 3. 创建用户
        self.repo.create(&input.into_new_user(hash)).await
    }
}
```

### 4.3 服务层测试示例

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use mockall::predicate::*;

    #[tokio::test]
    async fn register_new_user_success() {
        // Arrange: mock repo
        let mut mock_repo = MockUserRepo::new();
        mock_repo
            .expect_find_by_email()
            .with(eq("alice@example.com"))
            .returning(|_| Ok(None));  // 邮箱不存在 → 可以注册

        mock_repo
            .expect_create()
            .returning(|user| Ok(User {
                id: 1,
                email: user.email.clone(),
                username: user.username.clone(),
                password_hash: user.password_hash.clone(),
                ..Default::default()
            }));

        let service = AuthService { repo: mock_repo };

        // Act
        let result = service.register(RegisterInput {
            email: "alice@example.com".into(),
            username: "alice".into(),
            password: "secret123".into(),
            display_name: None,
        }).await;

        // Assert
        assert!(result.is_ok());
        let user = result.unwrap();
        assert_eq!(user.email, "alice@example.com");
        assert_eq!(user.username, "alice");
        // 密码不能是明文
        assert_ne!(user.password_hash, "secret123");
    }

    #[tokio::test]
    async fn register_duplicate_email_fails() {
        let mut mock_repo = MockUserRepo::new();
        mock_repo
            .expect_find_by_email()
            .with(eq("alice@example.com"))
            .returning(|_| Ok(Some(User::default())));  // 已存在

        let service = AuthService { repo: mock_repo };

        let result = service.register(RegisterInput {
            email: "alice@example.com".into(),
            username: "alice2".into(),
            password: "secret123".into(),
            display_name: None,
        }).await;

        assert!(matches!(result, Err(AppError::Conflict(_))));
    }
}
```

## 五、L3：API 集成测试

### 5.1 用 SQLite 替代 PostgreSQL

```rust
// tests/common/mod.rs

use sqlx::SqlitePool;

/// 创建测试用的 SQLite 内存数据库，运行 migration
pub async fn setup_test_db() -> SqlitePool {
    let pool = SqlitePool::connect("sqlite::memory:").await.unwrap();

    // 运行 migration（需要用 sqlx 的 migrate! 宏）
    sqlx::migrate!("./src/db/migrations")
        .run(&pool)
        .await
        .unwrap();

    pool
}

/// 创建完整的测试 App（Router + State）
pub async fn setup_test_app() -> (Router, SqlitePool) {
    let pool = setup_test_db().await;

    // Mock S3 和 Redis
    let s3 = MockS3Ops::new();
    let redis = None;  // 集成测试不依赖 Redis

    let state = AppState {
        pool: pool.clone(),
        s3: Arc::new(s3),
        redis: None,
        segmenter: Arc::new(init_segmenter()),
    };

    let app = build_routes(state);
    (app, pool)
}
```

### 5.2 API 集成测试示例

```rust
// tests/auth_test.rs

use axum_test::TestServer;
use common::setup_test_app;
use serde_json::json;

mod common;

#[tokio::test]
async fn register_and_login_flow() {
    let (app, _pool) = setup_test_app().await;
    let server = TestServer::new(app).unwrap();

    // Step 1: 注册
    let resp = server.post("/api/v1/auth/register")
        .json(&json!({
            "email": "test@example.com",
            "username": "testuser",
            "password": "secret123",
        }))
        .await;

    resp.assert_status_ok();
    let body: serde_json::Value = resp.json();
    assert!(body["token"].is_string());
    assert_eq!(body["username"], "testuser");

    // Step 2: 用返回的 token 获取用户信息
    let token = body["token"].as_str().unwrap();
    let resp = server.get("/api/v1/auth/me")
        .add_header("Authorization", format!("Bearer {token}"))
        .await;

    resp.assert_status_ok();
    let user: serde_json::Value = resp.json();
    assert_eq!(user["email"], "test@example.com");
}

#[tokio::test]
async fn login_with_wrong_password_returns_401() {
    let (app, _pool) = setup_test_app().await;
    let server = TestServer::new(app).unwrap();

    // 先注册
    server.post("/api/v1/auth/register")
        .json(&json!({
            "email": "test@example.com",
            "username": "testuser",
            "password": "secret123",
        }))
        .await;

    // 用错误密码登录
    let resp = server.post("/api/v1/auth/login")
        .json(&json!({
            "email": "test@example.com",
            "password": "wrong_password",
        }))
        .await;

    resp.assert_status(StatusCode::UNAUTHORIZED);
}

#[tokio::test]
async fn unauthenticated_request_returns_401() {
    let (app, _pool) = setup_test_app().await;
    let server = TestServer::new(app).unwrap();

    let resp = server.get("/api/v1/auth/me").await;
    resp.assert_status(StatusCode::UNAUTHORIZED);
}
```

async fn insert_and_find_document(pool: SqlitePool) {

## 六、AI 写测试的 Prompt 模板

### 6.1 单元测试 Prompt

```markdown
# 任务：为 {函数名} 写单元测试

## 位置
src/util/{文件名}.rs

## 被测函数签名
{粘贴函数签名和文档注释}

## 测试用例
1. 正常输入 → 预期输出
2. 边界情况 → 边界的输入和预期输出
3. 错误输入 → 预期报错类型

## 约束
- 用 #[test] 标注（非 async）或 #[tokio::test]（async）
- 不需要 mock（纯函数，无外部依赖）
- 测试放在文件底部 mod tests {} 里
- 每个测试函数 <20 行
```

### 6.2 服务层测试 Prompt

```markdown
# 任务：为 AuthService.register() 写测试

## 位置
src/service/auth_service.rs

## 被测函数签名
pub async fn register(&self, input: RegisterInput) -> Result<User, AppError>

## 依赖（需要 mock）
- UserRepo::find_by_email(email) → Option<User>
- UserRepo::create(new_user) → User

## 测试用例
1. 新用户注册成功 → 返回 User，password_hash 不是明文
2. 邮箱已存在 → 返回 AppError::Conflict
3. 密码太短 → 返回 AppError::BadRequest

## Mock 配置
- 用 mockall 自动生成的 MockUserRepo
- .expect_find_by_email() .with(eq("xxx")) .returning(...)
- .expect_create() .returning(...)

## 约束
- #[tokio::test] async
- Arrange / Act / Assert 三段式
- 放在文件底部 mod tests {}
```

### 6.3 API 集成测试 Prompt

```markdown
# 任务：为登录 API 写集成测试

## 端点
POST /api/v1/auth/login
Body: { "email": "...", "password": "..." }

## 预期行为
- 正确凭据 → 200 + { token, user }
- 错误密码 → 401
- 不存在邮箱 → 401（不区分，防枚举）

## 测试步骤
1. 调用 setup_test_app() 创建测试环境（已自动注入 SQLite + mock S3）
2. 先注册一个用户
3. 用不同凭据尝试登录

## 约束
- tests/auth_test.rs
- 用 axum_test::TestServer
- 用 .json() / .add_header() / .assert_status()
```
