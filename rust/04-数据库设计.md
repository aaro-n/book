# 数据库设计 - PostgreSQL 18 完整表结构

## 🎯 设计原则
1. **源文件保护** - 源PDF永远不修改，所有标记分离存储
2. **多语言支持** - 支持中日韩等非字母文字全文搜索
3. **性能优化** - 关键字段建索引，避免N+1查询
4. **审计追踪** - 关键操作记录历史日志（可选）
5. **数据一致性** - 外键约束确保参考完整性
6. **时间标准化** - 所有时间戳统一 UTC 存储（`TIMESTAMPTZ`），前端展示时转换为用户本地时区
---

## 📋 表结构设计

### 1. 用户表 - users

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    
    -- 认证信息
    email VARCHAR(255) UNIQUE NOT NULL,
    username VARCHAR(100) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    
    -- 用户信息
    display_name VARCHAR(100),
    avatar_url VARCHAR(500),
    
    -- 状态
    is_active BOOLEAN DEFAULT true,
    
    -- 时间戳（UTC）
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    last_login_at TIMESTAMPTZ
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_username ON users(username);
```

**说明**：
- email/username 都添加UNIQUE索引用于登录和注册验证
- password_hash 使用bcrypt加密存储（min 60字符）
- is_active 用于逻辑删除（不直接物理删除）

### 2. 文档表 - documents（源文件元数据）

```sql
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    
    -- 关联关系
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    
    -- 文档信息
    title VARCHAR(255) NOT NULL,
    description TEXT,
    
    -- 存储信息
    s3_key VARCHAR(500) NOT NULL,           -- S3完整路径
    mime_type VARCHAR(50) DEFAULT 'application/pdf',  -- application/pdf, application/epub+zip, etc.
    file_hash VARCHAR(64) NOT NULL,         -- SHA256哈希，防重复上传
    file_size_bytes BIGINT,                 -- 文件大小
    
    -- 内容信息
    page_count INTEGER,              -- 总页数
    language VARCHAR(10) DEFAULT 'auto',    -- 文档主要语言
    
  -- 元数据
    is_deleted BOOLEAN DEFAULT false,       -- 逻辑删除
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    
    -- 唯一约束：同一用户不能重复上传相同文件
    UNIQUE(user_id, file_hash)
);

CREATE INDEX idx_documents_user ON documents(user_id);
CREATE INDEX idx_documents_hash ON documents(file_hash);
CREATE INDEX idx_documents_created ON documents(user_id, created_at DESC);
```

**说明**：
- s3_key：S3中的完整路径，形如 `documents/user_123/doc_456/file.pdf`
- file_hash：用户上传相同PDF时可以检测出来，节省存储
- page_count：由后端在处理PDF时填充
- language：支持用户指定或自动检测（中文、日文、韩文等）
- 注意：**此表只存储元数据，不涉及任何修改操作**

### 3. 文档内容表 - document_content（用于搜索）

```sql
CREATE TABLE document_content (
    id SERIAL PRIMARY KEY,
    
    -- 关联
    document_id INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    
    -- 页码信息
    page_num INTEGER NOT NULL,
    
    -- 文本内容
    raw_text TEXT NOT NULL,         -- 原始文本
    
    -- 分词结果（应用层分词）
    segmented_text TEXT,            -- 分词后文本（空格分隔）
    
    -- PG 全文搜索 tsvector（计算列，自动维护，无需手动写入）
    content_tsv tsvector GENERATED ALWAYS AS 
        (to_tsvector('simple', coalesce(segmented_text, ''))) STORED,
    
    -- 时间戳（UTC）
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    
    -- 唯一约束：每文档每页只有一条记录
    UNIQUE(document_id, page_num)
);

-- 关键索引：GIN 倒排索引用于全文搜索（第 2 层搜索）
CREATE INDEX idx_content_document ON document_content(document_id, page_num);
CREATE INDEX idx_content_tsv ON document_content USING GIN (content_tsv);
```

**说明**：
- raw_text：原始文本，保持原样不修改（PDF 逐页提取，EPUB 逐章提取）
- page_num：PDF 为物理页码；EPUB 为章节序号（映射到 XHTML 文件）
- segmented_text：使用 jieba-rs 分词（CJK）或按空格切分（空格语言）后的结果，词之间用空格分隔
- content_tsv：PG 的 tsvector 计算列，由 `segmented_text` 自动生成。使用 `simple` 配置（文本已分好词，无需 PG 再分词）

**搜索查询示例**：

```sql
-- 第 1 层：LIKE 匹配
SELECT * FROM document_content
WHERE segmented_text LIKE '%阅读%' AND segmented_text LIKE '%PDF%';

-- 第 2 层：tsvector + GIN 索引，带相关性排序
SELECT *, ts_rank(content_tsv, query) AS rank
FROM document_content, to_tsquery('simple', '阅读 & PDF') AS query
WHERE content_tsv @@ query
ORDER BY rank DESC;
```

**示例数据**：
```json
{
  "raw_text": "我喜欢阅读PDF文档和笔记",
  "segmented_text": "我 喜欢 阅读 PDF 文档 笔记",
  "word_frequency": {"我": 1, "喜欢": 1, "阅读": 1, "PDF": 1},
  "word_positions": {
    "我": [[0, 1]],
    "喜欢": [[1, 3]],
    "阅读": [[3, 5]],
    "PDF": [[5, 8]],
    "文档": [[8, 10]],
    "笔记": [[11, 13]]
  }
}
```

### 4. 标注表 - annotations（统一存储，不修改源文件）

> **设计理念**：将所有标注（文字型、图形型、混合型）统一存于一张表。通过 `annotation_type` 字段区分类型，通过 `stroke_data` 字段存储图形数据。

#### 标注类型全景

```
所有标注
├── 文本型（可全文搜索）
│   ├─ highlight      高亮
│   ├─ underline      下划线
│   ├─ squiggly       波浪线
│   ├─ strikethrough  删除线
│   ├─ note           文字笔记
│   └─ stamp          图章（⭐重要 / ❓疑问 / ✅已读）
│
├── 图形型（不可搜索内容，可按页/颜色过滤）
│   ├─ freehand       手写/涂鸦
│   ├─ circle         画圈
│   ├─ rectangle      矩形框
│   ├─ line           直线/箭头
│   └─ polygon        多边形/折线
│
└── 混合型
    ├─ sticky_note    便签贴（图形 + 文字）
    ├─ freetext       自由文字框
    ├─ signature      签批
    ├─ link           文档内跳转链接
    └─ redact         遮盖/涂黑
```

#### 三阶段路线图

| 阶段 | 标注类型 | 覆盖 |
|------|---------|------|
| **Phase 1 MVP** | 高亮(多色)、下划线、波浪线、删除线、文字笔记、书签 | 90% 日常场景 |
| **Phase 2 进阶** | 手写/涂鸦、画圈/矩形/箭头、便签贴、图章 | iPad 笔记场景 |
| **Phase 3 完整** | 文字框、多边形、链接、遮盖/涂黑 | 审阅/校对场景 |

> **跨端兼容原则**：Flutter 客户端实现全部能力；Web 端可实现创建子集，但能**完整展示**客户端标记的所有类型（通过 SVG/Canvas 渲染图形数据）。

> **📌 建表请看 `23-数据库Schema映射.md`** — 本文档保留标注类型全景和设计理念；实际 DDL 以 23 的 W3C Web Annotation 兼容结构为准。
> 
> 设计理念简述：
> - `annotation_type` (18种) → 映射到 W3C `motivation` + `selector_type`
> - `bbox` → 存为 `locator_locations.fragments` 中的 `page=42&viewrect=x,y,w,h`
> - `stroke_data` → 存为 `locator_locations.other_locations.stroke`
> - `device_id` → 可空（非 Web 客户端才填），详见 `13-同步方案研究.md`
> 
> 图形型标注（freehand/circle/rectangle 等）的 SVG 数据通过 `locator.other_locations` 扩展字段存储，Web 端可渲染但不可编辑。

#### 18 种标注类型 ↔ W3C motivation 映射

| `annotation_type` | `motivation` | 坐标/图形存储位置 |
|---|---|---|
| highlight / underline / squiggly / strikethrough | `highlighting` | `locator_locations.fragments` (viewrect) |
| note / stamp / bookmark | `commenting` / `bookmarking` | `locator_locations.fragments` (page) |
| sticky_note / freetext / signature | `commenting` | `other_locations.stroke` (SVG) |
| freehand / circle / rectangle / line / polygon | `drawing` | `other_locations.stroke` (路径) |
| link / redact | `linking` / `assessing` | `other_locations.stroke` |

#### 可搜索性矩阵

| 标注类型 | 可搜索 | 搜索方式 |
|---------|:---:|------|
| highlight/underline/squiggly/strikethrough | ✅ | `segmented_text` LIKE / tsvector |
| note/sticky_note/freetext | ✅ | 同上 |
| stamp | ✅ | 图章文字可搜 |
| freehand/circle/rectangle/line/polygon | ❌ | 按 page_num + type 过滤 |
| link | ❌ | 按 page_num 过滤 |
| redact | ❌ | 不可见，仅管理端可见 |
| signature | ❌ | 按 page_num 过滤 |

#### Web Annotation JSON-LD 扩展示例

```json
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "type": "Annotation",
  "motivation": "drawing",
  "target": {
    "source": "/docs/101",
    "selector": {"type": "FragmentSelector", "value": "page=15"}
  },
  "body": [
    {"type": "TextualBody", "value": "这个公式重要", "format": "text/plain"},
    {"type": "Image", "value": "data:image/svg+xml;base64,...", "format": "image/svg+xml"}
  ],
  "style": {"type": "freehand", "color": "#FF0000", "strokeWidth": 2.0},
  "device": {"id": "urn:uuid:a1b2c3d4-..."},
  "modified": "2026-07-13T10:30:00Z"
}
```

#### Web 端渲染策略

```
Web 加载第 15 页:
  1. GET /api/v1/documents/{id}/annotations?page=15
  2. 拿到标注列表:
     ├─ 文本型 → CSS 绝对定位覆盖（半透明色块）
     ├─ 图形型 → 读取 stroke_data → Canvas/SVG 绘制
     └─ 混合型 → 图形 + 文字气泡
  3. Web 可编辑: 高亮、笔记、颜色修改
  4. Web 只读渲染: 手写、画圈、图章（用 SVG path）
```

### 5. 书签表 - bookmarks

> 书签是 `annotations` 表的简化特例（`annotation_type='bookmark'`，`text_body` 存标签名）。独立建表方便高频查询。

```sql
CREATE TABLE bookmarks (
    id SERIAL PRIMARY KEY,
    document_id INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    page_num INTEGER NOT NULL,
    label VARCHAR(100),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    CONSTRAINT unique_bookmark UNIQUE(document_id, user_id, page_num)
);

CREATE INDEX idx_bookmarks_doc_user ON bookmarks(document_id, user_id);
```

### 6. 阅读进度表 - reading_progress (OPDS Progression)

```sql
CREATE TABLE reading_progress (
    id SERIAL PRIMARY KEY,
    
    -- 关联关系
    document_id INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    
    -- 进度信息
    current_page INTEGER DEFAULT 1,
    total_pages INTEGER,
    read_percentage NUMERIC(5,2) DEFAULT 0,
    
    -- 设备追踪（用于多端同步冲突诊断，详见 13-同步方案研究.md）
    device_id VARCHAR(64),                   -- 哪台设备上报的进度
    device_name VARCHAR(100),                 -- 人类可读设备名
    
    -- 时间戳（UTC）
    last_read_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    
    -- 唯一约束：每用户每文档只有一条进度记录
    UNIQUE(document_id, user_id)
);

CREATE INDEX idx_progress_user ON reading_progress(user_id);
CREATE INDEX idx_progress_updated ON reading_progress(user_id, updated_at DESC);
```

**说明**：
- 记录用户的阅读位置
- last_read_at：用于"最近阅读"排序
- read_percentage：阅读进度百分比，便于显示进度条
- device_id/device_name：记录最近一次上报进度的设备，用于多端同步冲突诊断（详见 `24-同步策略对齐.md`）

### 7. S3 签名字典表 - presigned_urls

> 缓存 S3 预签名 URL，避免每次请求调 S3 SDK。详见 `16-S3存储方案.md`。

```sql
CREATE TABLE presigned_urls (
    s3_key VARCHAR(500) PRIMARY KEY,    -- S3 对象路径, e.g. "{doc_id}/pages/001.pdf"
    url TEXT NOT NULL,                   -- 完整签名 URL（含 CDN 域名 + 签名参数）
    expires_at TIMESTAMPTZ NOT NULL,     -- S3 签名过期时间（NOW() + 30天）
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_presigned_expires ON presigned_urls(expires_at);
```

**说明**：
- `s3_key`：S3 对象路径，与 `documents.s3_key` 指向不同（详见 `16-S3存储方案.md` 中的目录结构）
- `url`：完整的签名 URL，含 CDN 域名和所有 `X-Amz-*` 签名参数
- `expires_at`：签名真实过期时间，每天定时清理过期行
- 此表是签名 URL 的 L2 缓存（L1 = Redis），Redis 不可用时由此表保底

### 查询用户的所有高亮

```sql
SELECT a.* 
FROM annotations a
WHERE a.user_id = 123 
  AND a.document_id = 456
  AND a.is_deleted = false
ORDER BY a.page_num, a.created_at;
```

### 全文搜索用户的标记

不需要专门的 `searchable_marks` 表，使用 `UNION ALL` 联合查询即可满足中小型应用需求：

```sql
SELECT 'content' as type, id, page_num, raw_text as text, ts_rank(content_tsv, query) as score
FROM document_content, to_tsquery('simple', '人工 & 智能') query
WHERE document_id = 456 AND content_tsv @@ query

UNION ALL

SELECT 'annotation' as type, id, page_num, text_body as text, ts_rank(to_tsvector('simple', coalesce(segmented_text, '')), query) as score
FROM annotations, to_tsquery('simple', '人工 & 智能') query
WHERE document_id = 456 AND user_id = 123 AND is_deleted = false
  AND to_tsvector('simple', coalesce(segmented_text, '')) @@ query

ORDER BY score DESC LIMIT 20;
```

### 导出用户的标记为Markdown

```sql
SELECT 
    d.title,
    a.page_num,
    a.text_body,
    a.annotation_type,
    a.created_at
FROM annotations a
JOIN documents d ON a.document_id = d.id
WHERE a.user_id = 123
  AND a.is_deleted = false
ORDER BY a.page_num, a.created_at;
```

### 检测重复上传

```sql
SELECT * FROM documents 
WHERE user_id = 123 
  AND file_hash = 'abc123...'
  AND is_deleted = false;
```

## 📊 性能优化要点

1. **索引策略**
   - 外键字段都要索引（user_id, document_id）
   - 查询条件的前缀列要索引
   - GIN索引用于JSONB和tsvector

2. **避免N+1查询**
   ```sql
   -- ❌ 避免：循环查询
   SELECT * FROM documents WHERE user_id = 123;
   for each doc:
       SELECT * FROM annotations WHERE document_id = doc.id;
   
   -- ✅ 推荐：一次JOIN
   SELECT a.* FROM annotations a
   JOIN documents d ON a.document_id = d.id
   WHERE d.user_id = 123;
   ```

3. **连接池**
   - max_connections: 100（足够用）
   - idle_in_transaction_session_timeout: 10分钟（自动断开空闲)

4. **定期维护**
   ```sql
   -- 每周运行：清理WAL日志
   VACUUM FULL;
   REINDEX;
   
   -- 每天运行：自动清理
   AUTOVACUUM = on;
   ```

---

## 🔐 安全考虑

1. **数据隔离** - 所有查询都带user_id过滤
2. **物理删除 vs 逻辑删除** - documents用逻辑删除（is_deleted）
3. **外键约束** - ON DELETE CASCADE确保数据一致
4. **唯一约束** - 防止重复数据

---
## 总结

这个数据库设计确保：
- ✅ **源文件安全** - PDF永不修改
- ✅ **多语言搜索** - 应用层分词，PG存储
- ✅ **高性能** - 合理的索引和表设计
- ✅ **数据完整** - 外键约束和唯一性保证
- ✅ **易于扩展** - 表结构清晰，易添加新功能


### 10. 自定义词典表 - custom_dictionary

用于存储应用层分词（jieba-rs）的动态扩展词汇，支持热更新。

```sql
CREATE TABLE custom_dictionary (
    id SERIAL PRIMARY KEY,
    
    -- 词汇信息
    term VARCHAR(100) NOT NULL UNIQUE,
    frequency INT DEFAULT 0,
    nature VARCHAR(20) DEFAULT 'nz',
    
    -- 状态
    is_enable BOOLEAN DEFAULT TRUE,
    
    -- 时间戳（UTC）
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_dict_term ON custom_dictionary(term);
CREATE INDEX idx_dict_enable ON custom_dictionary(is_enable);
```
