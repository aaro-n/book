# 认证与安全 — Session Token + API Key 极简方案

> **设计原则**：学 Komga，不学银行。个人书库不需要 JWT access/refresh 双 token 轮换。
> **参考基准**：Kavita (C# JWT)、Komga (Kotlin Session)、Calibre-Web (Python Session Cookie) 的实战分析。

---

## 一、三大项目认证方案对比

| | Kavita (C#) | Komga (Kotlin) | Calibre-Web (Python) |
|---|---|---|---|
| **主认证** | JWT (10天) | HTTP Session (Caffeine 缓存) | Flask Session Cookie |
| **刷新** | 一次性 Refresh Token | Session 自动续期 | Cookie + 31天续期 |
| **API Key** | ✅ AuthKey（多把，可选过期） | ✅ ApiKey（UUID，哈希存） | ❌ |
| **设备管理** | ✅ EmailDevice + ClientDevice | ❌ | ❌ |
| **OAuth/OIDC** | ✅ | ✅ | ✅ |
| **OPDS 接入** | AuthKey 放 URL 路径 | API Key 放 Header | HTTP Basic Auth |
| **记住我** | JWT 10天 | 365天 Cookie | 365天 Cookie |

### 关键发现

1. **没有人搞复杂的 token 刷新** — Kavita 是唯一用 JWT + Refresh 的，但 JWT 有效期 10 天，不是 15 分钟。个人书库不是银行。

2. **Komga 最简单最稳** — 老派 Session Cookie，不用 JWT，不用 refresh。`KOMGA-SESSION` cookie + Caffeine 缓存。客户端用 `X-Auth-Token` header。

3. **三个人都搞了 API Key** — 书库软件标配，给 OPDS 客户端、Kobo 设备、脚本访问。Komga 的做法最干净：UUID 去横线 → 哈希存储 → `X-API-Key` header。

4. **设备管理只有 Kavita 做了** — 另两家不管"哪台设备在登录"。Kavita 的 ClientDevice 只是统计用途，不参与认证流程。

## 二、本项目方案：Session Token + API Key

### 2.1 认证流程全景

```
┌─────────────────────────────────────────────────────────────┐
│                       认证入口                                │
│                                                             │
│  POST /api/v1/auth/login                                    │
│    email + password                                          │
│         │                                                    │
│         ▼                                                    │
│  ┌─────────────────┐                                        │
│  │ bcrypt 验证密码   │                                        │
│  └────────┬────────┘                                        │
│           │ ✅ 通过                                          │
│           ▼                                                  │
│  ┌─────────────────┐                                        │
│  │ 生成 Session Token│  ← 32 字节随机数，Base64URL 编码        │
│  │ 存入 Redis        │  ← key: "session:{token}"             │
│  │ TTL: 7 天         │  ← value: {user_id, username, role}   │
│  └────────┬────────┘                                        │
│           │                                                  │
│           ▼                                                  │
│  返回给客户端：                                               │
│  ┌──────────────────────────────────────────────────┐       │
│  │ Web:  Set-Cookie: SESSION=xxx; HttpOnly; Secure   │       │
│  │ Flutter: { "token": "xxx" } → flutter_secure_storage│     │
│  │ API:   Header: Authorization: Bearer xxx          │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
│  后续请求鉴权：                                               │
│  ┌──────────────────────────────────────────────────┐       │
│  │ 中间件提取 token → Redis GET "session:{token}"    │       │
│  │   → 命中：注入 UserContext，放行                   │       │
│  │   → 未命中：401 Unauthorized                       │       │
│  └──────────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 为什么不用 JWT？

| 问题 | JWT 方案 | Session 方案 |
|------|---------|-------------|
| **服务端作废** | 需要黑名单或等过期 | 删 Redis key 即刻生效 |
| **Token 大小** | ~200-500 bytes（含签名+claims） | ~43 chars（Base64URL 32字节） |
| **刷新逻辑** | 需要 refresh token + 竞态处理 | 不需要，每次请求可续期 TTL |
| **代码量** | ~400 行（签发+验证+刷新+黑名单） | ~150 行 |
| **客户端复杂度** | 需处理 401 → refresh → 重试 | 只管存 token，401 = 重新登录 |
| **调试** | 需解码 JWT 看内容 | `redis-cli GET session:xxx` 直接看 |

> **唯一劣势**：Session 方案依赖 Redis。但本项目架构图里本来就有 Redis（缓存/搜索），不增加新依赖。

### 2.3 Session 数据结构

**Redis 中存储**（key: `session:{token}`, TTL: 7天）：

```json
{
  "user_id": 1,
  "username": "alice",
  "role": "user"
}
```

> 只存最小必要信息。中间件鉴权后注入 `UserContext`，后续 handler 直接从 context 取，无需每次查 PG。

**UserContext**（Rust 侧）：

```rust
#[derive(Clone, Debug)]
pub struct UserContext {
    pub user_id: i32,
    pub username: String,
    pub role: UserRole,
}

#[derive(Clone, Debug, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "lowercase")]
pub enum UserRole {
    User,
    Admin,
}
```

### 2.4 Token 生成

```rust
use rand::Rng;
use base64::{Engine as _, engine::general_purpose::URL_SAFE_NO_PAD};

fn generate_session_token() -> String {
    let mut bytes = [0u8; 32];
    rand::thread_rng().fill(&mut bytes);
    URL_SAFE_NO_PAD.encode(bytes)  // 43 chars, e.g. "kG7xPq2vWm9zL5nR8tY1aB4cD6eF0hJ3xK..."
}
```

> 32 字节随机数 → Base64URL 无填充 → 43 字符。128 位熵，暴力破解不可行。

## 三、API Key 方案

### 3.1 设计

```
用户后台生成 API Key
        │
        ▼
  生成 UUID v4 → 去横线 → 32 字符明文字符串
        │
        ▼
  SHA256(明文) → 64 字符哈希 → 存入 PG api_keys 表
        │
        ▼
  明文仅展示一次给用户："保存好，不会再显示"
```

**使用方式**：

| 场景 | 方式 | 示例 |
|------|------|------|
| **OPDS 客户端** | URL 路径 | `/opds/{apiKey}/catalog` |
| **API 脚本** | Header | `X-API-Key: {apiKey}` |
| **Kobo 设备** | URL 路径（兼容 Komga） | `/kobo/{apiKey}/v1/library/...` |

> OPDS 路径放 API Key 是因为很多 OPDS 客户端（KOReader、FBReader）不支持自定义 Header，只支持填 URL。

### 3.2 数据库表

```sql
CREATE TABLE api_keys (
    id SERIAL PRIMARY KEY,

    -- 关联
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- 密钥
    key_hash VARCHAR(64) NOT NULL UNIQUE,   -- SHA256(明文)
    key_prefix VARCHAR(8) NOT NULL,          -- 前 8 位，用于界面展示 "a1b2c3d4****"

    -- 元数据
    name VARCHAR(100) NOT NULL,              -- 用户命名："我的 KOReader"、"备份脚本"
    last_used_at TIMESTAMPTZ,               -- 上次使用时间

    -- 可选过期
    expires_at TIMESTAMPTZ,                  -- NULL = 永不过期

    -- 状态
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_apikeys_user ON api_keys(user_id);
CREATE INDEX idx_apikeys_hash ON api_keys(key_hash);
```

### 3.3 中间件鉴权优先级

```
请求进入
  │
  ├─ Cookie / Authorization Header 有 Session Token？
  │    → Redis 验证 → 注入 UserContext → 放行
  │
  ├─ Header: X-API-Key 存在？
  │    → SHA256 → 查 api_keys 表 → 注入 UserContext → 放行
  │
  ├─ URL 路径含 /opds/{maybe_api_key}/...？
  │    → SHA256 → 查 api_keys 表 → 注入 UserContext → 放行
  │
  └─ 都没有 → 401 Unauthorized
```

> Session Token 优先于 API Key。同一请求两者都带时，用 Session。

## 四、API 端点定义

### 4.1 注册

- **POST** `/api/v1/auth/register`
- **Body**:
```json
{
  "email": "alice@example.com",
  "username": "alice",
  "password": "mySecret123",
  "display_name": "Alice"
}
```
- **验证规则**:
  - email 格式合法、唯一
  - username 3-30 字符、字母数字下划线、唯一
  - password ≥ 8 字符、含字母和数字
- **Response** `201 Created`:
```json
{
  "user_id": 1,
  "username": "alice",
  "token": "kG7xPq2vWm9zL5nR8tY1aB4cD6eF0hJ3xK..."
}
```
- **Response Headers**: `Set-Cookie: SESSION=xxx; HttpOnly; Secure; SameSite=Lax; Max-Age=604800; Path=/`

> 注册即登录，直接返回 token。MVP 不需要邮箱验证。

### 4.2 登录

- **POST** `/api/v1/auth/login`
- **Body**:
```json
{
  "email": "alice@example.com",
  "password": "mySecret123"
}
```
- **Response** `200 OK`:
```json
{
  "token": "kG7xPq2vWm9zL5nR8tY1aB4cD6eF0hJ3xK...",
  "user": {
    "id": 1,
    "username": "alice",
    "display_name": "Alice",
    "role": "user"
  }
}
```
- **Response Headers**: 同注册，Set-Cookie
- **错误** `401`: `{"error": "invalid_credentials", "message": "邮箱或密码错误"}`

> 不区分"用户不存在"和"密码错误"，防止邮箱枚举。

### 4.3 登出

- **POST** `/api/v1/auth/logout`
- **Auth**: Session Token
- **Response** `200 OK`:
```json
{ "status": "logged_out" }
```
- **Response Headers**: `Set-Cookie: SESSION=; Max-Age=0`（清除 cookie）
- **服务端**: `DEL session:{token}`

### 4.4 获取当前用户

- **GET** `/api/v1/auth/me`
- **Auth**: Session Token
- **Response** `200 OK`:
```json
{
  "id": 1,
  "username": "alice",
  "display_name": "Alice",
  "email": "alice@example.com",
  "role": "user",
  "created_at": "2026-07-01T10:00:00Z",
  "last_login_at": "2026-07-13T08:30:00Z"
}
```

### 4.5 修改密码

- **PUT** `/api/v1/auth/password`
- **Auth**: Session Token
- **Body**:
```json
{
  "old_password": "mySecret123",
  "new_password": "newSecret456"
}
```
- **Response** `200 OK`: `{"status": "password_changed"}`
- **副作用**: 删除该用户所有 Session（所有设备需重新登录）

### 4.6 密码重置（MVP 可选）

> MVP 阶段不实现邮件重置。如需重置，管理员直接在数据库改（或提供管理员 CLI）。
> Phase 2 可以加入邮件验证流程。

### 4.7 API Key 管理

| 方法 | 路径 | 说明 |
|------|------|------|
| **GET** | `/api/v1/auth/api-keys` | 列出当前用户的所有 API Key（只显示 name + key_prefix + 时间） |
| **POST** | `/api/v1/auth/api-keys` | 创建新 API Key |

## 五、中间件实现

### 5.1 Session 鉴权中间件

```rust
// src/middleware/auth.rs

use axum::{
    extract::{Request, State},
    http::{header, StatusCode},
    middleware::Next,
    response::Response,
};
use crate::{AppState, models::UserContext, util::errors::AppError};

/// 从 Cookie / Authorization Header 提取 Session Token，Redis 验证后注入 UserContext
pub async fn require_auth(
    State(state): State<AppState>,
    mut req: Request,
    next: Next,
) -> Result<Response, AppError> {
    let token = extract_session_token(&req)?;

    let session_key = format!("session:{token}");
    let user_json: String = state.redis.get(&session_key).await
        .map_err(|_| AppError::Unauthorized("未登录或会话已过期".into()))?;

    let user: UserContext = serde_json::from_str(&user_json)
        .map_err(|_| AppError::Unauthorized("会话数据异常".into()))?;

    // 续期：每次请求刷新 TTL
    let _ = state.redis.expire(&session_key, 604800).await; // 7 天

    req.extensions_mut().insert(user);
    Ok(next.run(req).await)
}

fn extract_session_token(req: &Request) -> Result<String, AppError> {
    // 1. Cookie 优先
    if let Some(cookie_header) = req.headers().get(header::COOKIE) {
        if let Some(token) = parse_cookie(cookie_header, "SESSION") {
            return Ok(token);
        }
    }

    // 2. Authorization: Bearer xxx
    if let Some(auth_header) = req.headers().get(header::AUTHORIZATION) {
        if let Ok(value) = auth_header.to_str() {
            if let Some(token) = value.strip_prefix("Bearer ") {
                return Ok(token.to_string());
            }
        }
    }

    Err(AppError::Unauthorized("缺少认证凭据".into()))
}

fn parse_cookie(header: &HeaderValue, name: &str) -> Option<String> {
    let cookie_str = header.to_str().ok()?;
    for part in cookie_str.split(';') {
        let mut kv = part.trim().splitn(2, '=');
        if kv.next()? == name {
            return kv.next().map(String::from);
        }
    }
    None
}
```

### 5.2 API Key 鉴权中间件

```rust
/// 从 X-API-Key Header 或 URL 路径提取 API Key，SHA256 后查数据库
pub async fn require_api_key(
    State(state): State<AppState>,
    mut req: Request,
    next: Next,
) -> Result<Response, AppError> {
    let api_key = extract_api_key(&req)?;

    let hash = sha256_hex(&api_key);
    let user = sqlx::query_as!(
        UserContext,
        r#"SELECT u.id as user_id, u.username, 'user' as "role: UserRole"
           FROM api_keys k JOIN users u ON k.user_id = u.id
           WHERE k.key_hash = $1 AND k.is_active = true
             AND (k.expires_at IS NULL OR k.expires_at > NOW())
             AND u.is_active = true"#,
        hash
    )
    .fetch_optional(&state.pool)
    .await?
    .ok_or(AppError::Unauthorized("无效的 API Key".into()))?;

    // 更新 last_used_at
    let _ = sqlx::query!(
        "UPDATE api_keys SET last_used_at = NOW() WHERE key_hash = $1",
        hash
    )
    .execute(&state.pool)
    .await;

    req.extensions_mut().insert(user);
    Ok(next.run(req).await)
}

fn extract_api_key(req: &Request) -> Result<String, AppError> {
    // 1. X-API-Key Header
    if let Some(value) = req.headers().get("X-API-Key") {
        return Ok(value.to_str()?.to_string());
    }

    // 2. URL 路径: /opds/{apiKey}/...
    let path = req.uri().path();
    if let Some(rest) = path.strip_prefix("/opds/") {
        if let Some(key) = rest.split('/').next() {
            if key.len() == 32 && key.chars().all(|c| c.is_ascii_hexdigit()) {
                return Ok(key.to_string());
            }
        }
    }

    Err(AppError::Unauthorized("缺少 API Key".into()))
}

fn sha256_hex(input: &str) -> String {
    use sha2::{Sha256, Digest};
    let mut hasher = Sha256::new();
    hasher.update(input.as_bytes());
    hex::encode(hasher.finalize())
}
```

### 5.3 可选鉴权中间件（用于 OPDS 公开路由）

```rust
/// 尝试鉴权，但不强制。用于 OPDS 路由——有 token 返回个性化目录，没 token 返回公开目录
pub async fn optional_auth(
    State(state): State<AppState>,
    mut req: Request,
    next: Next,
) -> Result<Response, AppError> {
    // 尝试 Session Token
    if let Ok(token) = extract_session_token(&req) {
        if let Ok(user_json) = state.redis.get::<_, String>(&format!("session:{token}")).await {
            if let Ok(user) = serde_json::from_str::<UserContext>(&user_json) {
                req.extensions_mut().insert(user);
            }
        }
    }
    // 尝试 API Key
    else if let Ok(api_key) = extract_api_key(&req) {
        // ... 同 require_api_key 逻辑
    }
    // 都没有也不报错，走公开逻辑

    Ok(next.run(req).await)
}
```

## 六、密码安全

### 6.1 哈希策略

```rust
// 使用 bcrypt, cost = 12
use bcrypt::{hash, verify, DEFAULT_COST};

const BCRYPT_COST: u32 = 12; // ~250ms on modern CPU, 兼顾安全与体验

pub fn hash_password(password: &str) -> Result<String, AppError> {
    hash(password, BCRYPT_COST).map_err(|e| AppError::Internal(e.into()))
}

pub fn verify_password(password: &str, hash: &str) -> Result<bool, AppError> {
    verify(password, hash).map_err(|e| AppError::Internal(e.into()))
}
```

| cost | 耗时 | 适用 |
|------|------|------|
| 10 | ~100ms | 高并发 |
| **12** | **~250ms** | **个人服务，平衡安全与体验** |
| 14 | ~1s | 高安全 |

### 6.2 密码强度规则

```rust
pub fn validate_password(password: &str) -> Result<(), AppError> {
    if password.len() < 8 {
        return Err(AppError::BadRequest("密码至少 8 个字符".into()));
    }
    if !password.chars().any(|c| c.is_ascii_alphabetic()) {
        return Err(AppError::BadRequest("密码必须包含字母".into()));
    }
    if !password.chars().any(|c| c.is_ascii_digit()) {
        return Err(AppError::BadRequest("密码必须包含数字".into()));
    }
    Ok(())
}
```

## 七、会话管理策略

### 7.1 Redis Key 设计

| Key | Value | TTL | 说明 |
|-----|-------|-----|------|
| `session:{token}` | `{"user_id":1,"username":"alice","role":"user"}` | 7天 | 主会话 |
| `login_attempts:{ip}` | `5` | 15分钟 | 登录限流计数器 |

### 7.2 会话生命周期

```
创建：登录/注册成功 → Redis SET + TTL 7天
续期：每次验证通过 → Redis EXPIRE 重置为 7天
失效：登出 → Redis DEL；修改密码 → 删该用户所有 session
自动过期：7天无操作 → Redis 自动删除
```

### 7.3 修改密码时批量踢出

```rust
pub async fn delete_user_sessions(
    redis: &mut ConnectionManager,
    user_id: i32,
) -> Result<(), AppError> {
    // 扫描所有 session:* key，过滤出该 user_id 的并删除
    // 或者：Redis 存一个反向索引 set "user_sessions:{user_id}" → {token1, token2, ...}
    let user_sessions_key = format!("user_sessions:{user_id}");
    let tokens: Vec<String> = redis.smembers(&user_sessions_key).await?;

    for token in &tokens {
        let _: () = redis.del(&format!("session:{token}")).await?;
    }
    let _: () = redis.del(&user_sessions_key).await?;

    Ok(())
}
```

> 登录时额外执行 `SADD user_sessions:{user_id} {token}` + `EXPIRE user_sessions:{user_id} 604800`，建立反向索引。

## 八、OPDS 路由认证适配

原有的 OPDS 路由中 `Authorization: Bearer <token>` 保持不变，但 token 从 JWT 换为 Session Token。同时新增 API Key 鉴权路径：

```
OPDS 路由鉴权：
  ├─ /opds/catalog              → optional_auth（公开也可见，有 token 则个性化）
  ├─ /opds/{apiKey}/catalog     → require_api_key（URL 路径带 API Key）
  ├─ /opds/catalog/search       → require_auth（搜索需登录）
  ├─ /opds/acquisition/{id}     → require_auth（下载需登录）
  └─ /opds/v2/*                 → 同上
```

> `/opds/{apiKey}/catalog` 专为 KOReader 等不支持自定义 Header 的客户端设计。API Key 放在 URL 路径中。

```sql
    key_prefix VARCHAR(8) NOT NULL,
    name VARCHAR(100) NOT NULL,
    last_used_at TIMESTAMPTZ,
    expires_at TIMESTAMPTZ,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_apikeys_user ON api_keys(user_id);
CREATE INDEX idx_apikeys_hash ON api_keys(key_hash);
```

`users` 表无需变更。原 `password_hash` 字段继续使用（bcrypt）。

## 十一、不需要的东西（明确排除）

| 排除项 | 理由 |
|--------|------|
| ❌ JWT access/refresh 双 token | 个人书库不需要 15 分钟过期，代码量 3 倍 |
| ❌ Token 轮换 (rotation) | Session 方案天然支持随时作废 |
| ❌ 设备管理表 (devices) | Komga 和 Calibre-Web 都不做，Kavita 做了也只是统计 |
| ❌ OAuth/OIDC（MVP） | 先不做，Phase 2 可选 |
| ❌ 邮箱验证（MVP） | 个人部署不跑邮件服务，Phase 2 可选 |
| ❌ 记住我 (Remember Me) | Session 本身 7 天，够用 |
