## 🔄 数据流设计

### 1. 文档上传流程

```
用户选择PDF文件
    ↓
前端验证（大小、格式）
    ↓
上传到后端
    ↓
后端验证文件完整性
    ↓
保存到S3存储
    ↓
计算SHA256哈希（去重）
    ↓
提取文本并分词
    ↓
┌────────────────────┐
│ 存储处理结果              │
├──────────────────────┤
│ 1. PG - document_content表      │
│    - raw_text: 原文本           │
│    - segmented_text: 分词结果   │
│    - word_frequency: 词频统计   │
│                            │
│ 2. 搜索引擎（异步）             │
│    - ES或Meilisearch        │
│    - 自动处理分词和索引         │
└──────────────────────┘
    ↓
更新文档元数据
    ↓
返回用户：上传成功
```

### 2. 用户标记流程

```
用户在阅读器中操作
    ↓
┌─────────────────┐
│ 选择操作类型           │
├────────────────────────┤
│ 高亮文本 → highlights表       │
│ 添加注释 → annotations表      │
│ 书签页面 → bookmarks表        │
└──────────────┘
    ↓
分词用户输入
    ↓
存储到PG
    ↓
异步索引到搜索引擎
    ↓
返回用户：操作成功
    ↓
源PDF文件保持不变（关键！）
```

### 3. 搜索流程

```
用户输入搜索词
    ↓
应用层分词
    ↓
┌─────────────────────┐
│ 查询搜索引擎（优先级）          │
├─────────────────────┤
│ 1. 尝试主搜索引擎（ES默认）    │
│    ├─ 命中 ✓ → 返回结果        │
│    └─ 失败 → 继续              │
│                 │
│ 2. 尝试备选搜索引擎     │
│    ├─ 命中 ✓ → 返回结果        │
│    └─ 失败 → 继续              │
│                    │
│ 3. 降级到PG搜索（最后手段）   │
│    └─ ILIKE查询（简单但可用）  │
└───────────────────────┘
    ↓
高亮匹配词
    ↓
返回结果给用户
```

## 💾 内存管理策略

### 启动时内存分布（512MB限制）

```
基础框架              ~8MB      ← Rust 显著优势
├─ Axum + Tower    3MB
├─ AWS SDK (Rust)  3MB
└─ 其他库       2MB

分词引擎          ~25MB
├─ jieba-rs 词库     15MB
├─ 自定义词库         5MB
└─ 缓存            5MB

数据访问层            ~15MB
├─ sqlx 连接池       5MB
├─ redis-rs 连接     3MB
└─ 序列化缓冲        7MB

运行时开销            ~30MB
├─ Tokio 运行时      8MB
├─ 堆分配           15MB
└─ 缓冲区            7MB

═══════════════════
总计               ~78MB（有 434MB 余量！）

对比 Go 版 ~340MB，Rust 内存占用仅为 Go 的 **23%**
```

### 内存优势来源

| 优势 | 效果 | 原理 |
|------|------|------|
| **无 GC** | -60MB 堆 | 没有 GC 元数据开销，RAII 立即释放 |
| **枚举紧凑** | -15MB | `Option<Box<T>>` 编译为单指针 |
| **无 Goroutine 栈** | -30MB | Tokio task 初始 ~64 bytes（无栈协程） |
| **静态分发** | -10MB | 泛型单态化，无虚表 |

### 内存优化技巧（几乎不需要！）

| 优化项 | 效果 | 实现方式 |
|-----|------|---|
| **jieba-rs 精简词典** | -10MB | 仅加载高频词（可选） |
| **LRU缓存** | -5MB | 限制缓存条数 |
| **零拷贝解析** | 省 CPU | serde borrow 模式 |

## 🔐 安全考虑

### 认证和授权

1. **用户认证** - JWT token
```rust
// 流程：Login → 生成Token → 存入Redis → 后续请求验证
// Token过期：1小时
// Refresh Token：7天
```

2. **权限验证** - 用户隔离
   ```rust
   // Axum middleware extractor 从 JWT 提取 user_id
   // 所有操作都验证 user_id
   // 确保用户只能访问自己的文档和标记
   // 数据库约束：UNIQUE(user_id, document_id)
   ```

### 数据保护

1. **源文件不可修改**
   ```sql
   -- PDF文件完全由用户标记表控制
   -- 源PDF永远存储在S3，不动
   -- 标记表有CREATE/UPDATE/DELETE权限
   ```

2. **密码安全**
   ```rust
   // 使用 bcrypt crate 哈希，不存储明文
   // 密码验证比对时间恒定（防时序攻击）
   ```

3. **传输安全**
   ```
   -- HTTPS传输
   -- CORS配置严格
   -- 敏感操作需要CSRF token
   ```

## 📊 性能指标

### 目标指标

| 指标 | 目标 | 说明 |
|------|---|
| **API延迟** | <200ms | 99%请求 |
| **分词速度** | <100ms | 1000词以内 |
| **搜索响应** | <300ms | 包含排序和分页 |
| **文件上传** | <5s | 100MB文件 |
| **内存占用** | <512MB | Go应用专用 |
| **并发连接** | >100 | 同时在线用户 |

### 性能优化手段

1. **分词引擎选择**
   - 默认GSE（快速轻量）
   - 必要时启用Jieba（精准）
   - 结果缓存避免重复分词

2. **数据库优化**
   - 索引优化（GIN索引for tsvector）
   - 连接池管理
   - 慢查询监控

3. **搜索优化**
   - 搜索结果缓存
   - 热词预加载
   - 索引合并（定期）

4. **缓存策略**
   - Redis缓存热点数据
   - LRU缓存本地热词
   - 缓存穿透保护

## 📈 监控和日志

### 关键指标监控

```
应用层
├─ API请求数/秒
├─ 错误率（%）
├─ 响应时间分布
└─ 分词引擎使用频率

数据库层
├─ 查询响应时间
├─ 慢查询日志

├─ 搜索延迟
├─ 索引大小
├─ 查询成功率
└─ 回源请求率
```

### 日志策略

```go
// 日志级别
DEBUG   - 开发调试信息（本地开发）
INFO    - 关键业务事件（上传、搜索）
WARN    - 警告但不影响（降级、重试）
ERROR   - 错误但可恢复（连接失败）
FATAL   - 致命错误（启动失败）

// 日志内容
- 请求ID（trace_id）追踪整个请求链路
- 用户ID隔离用户数据
- 操作耗时（milliseconds）
- 错误堆栈（仅FATAL/ERROR）
```

- **简洁清晰** - 架构易理解和维护
