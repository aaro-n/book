## 🧩 三组件架构

系统由**三个独立项目**组成，各司其职，互不依赖：

```
┌──────────────────────────────────────────────────────────┐
│                    服务端 (Rust/Axum)                      │
│  512MB VPS · 位置: book-server                            │
│                                                          │
│  只做轻活：                                                │
│  ├─ 用户认证 / 文档管理 / 搜索 API                          │
│  ├─ 轻量 PDF 提取 (lopdf, 仅文字版, <200MB)                │
│  ├─ jieba-rs 分词 (CJK) + 空格切分 (非 CJK)               │
│  ├─ 接收 .bookpkg 压缩包，解包直接入库                       │
│  └─ 按页分发页面图片/PDF (S3 静态文件)                       │
│                                                          │
│  不做：OCR / EPUB 解析 / 大文件 / 重型处理                   │
└──────────────┬────────────────────┬───────────────────────┘
               │                    │
      REST API │                    │ 上传 .bookpkg
               ▼                    ▼
┌──────────────────────┐  ┌────────────────────────────────┐
│  阅读器客户端           │  │  导入 CLI (独立 Rust 工具)        │
│  Flutter + Rust       │  │  位置: book-import              │
│                       │  │                                │
│  ├─ PDF 阅读           │  │  完全独立，不共享服务端/客户端代码  │
│  ├─ 高亮/书签/笔记      │  │                                │
│  ├─ 本地离线搜索        │  │  ├─ OCR 扫描件 (Tesseract)      │
│  ├─ 与服务端同步        │  │  ├─ EPUB/MOBI 解析 + 分词       │
│  ├─ 下载整本 PDF 离线读  │  │  ├─ 重型 PDF 渲染 (大文件/特殊语言)│
│  └─ 日常阅读使用        │  │  ├─ jieba-rs 分词 (所有语言)     │

### 服务端处理能力边界

服务端处理能力取决于三个因素：**格式、文字层、语言、文件大小**。不满足条件的统一提示用户使用 CLI 工具。

| 条件 | 谁处理 | 原因 |
|------|--------|------|
| **PDF**: 文字版 + CJK/空格语言 + < 200MB | **服务端** | `lopdf` 提取文字，内存 ~60MB |
| **PDF**: 扫描件（无文字层） | **导入 CLI** | 需要 Tesseract OCR（200-400MB） |
| **PDF**: > 200MB | **导入 CLI** | 服务端内存扛不住 |
| **PDF**: 特殊语言（泰语/阿拉伯语等） | **导入 CLI** | 服务端不装对应分词 |
| **EPUB**: CJK/空格语言 + < 50MB | **服务端** | 解压 + 去标签 + jieba-rs，内存 ~20MB |
| **EPUB**: 特殊语言 / 超大 / 大量图片 | **导入 CLI** | 需对应分词 + 图片渲染 |
| **MOBI** | **导入 CLI** | 服务端不做格式转换 |

检测方法：服务端用 `lopdf` 读前 3 页，检查是否有文本对象。无文字层 → 返回 `PdfType::Scanned` → 提示用户用 CLI。

### jieba-rs 懒加载

jieba-rs 编译进服务端二进制，非独立进程。采用懒加载策略：

```rust
// 首次使用时加载词典 (~15-30MB)，10分钟无新任务后释放
fn get_segmenter() -> Jieba { /* lazy init */ }
fn drop_segmenter() { /* 释放内存 */ }
```

**空闲时内存占用几乎为 0。** 只在处理 PDF/EPUB 的那几十秒占用 15-30MB。

### 服务端语言支持

服务端只处理擅长的语言，CLI 覆盖其余所有：

```rust
fn is_supported_lang(lang: &str) -> bool {
    // CJK → jieba-rs；空格语言 → split_whitespace
    matches!(lang, "zh" | "ja" | "ko"
             | "en" | "fr" | "de" | "ru" | "es" | "pt"
             | "it" | "nl" | "sv" | "da" | "no" | "fi"
             | "tr" | "vi" | "id" | "ms" | "pl")
}
```

> 泰语、阿拉伯语、高棉语等 → 提示用户用 CLI 预处理后上传 .bookpkg。CLI 工具用三策略（jieba-rs / 空格 / Unicode 单词边界）覆盖全语言。

## 📄 文档格式处理

### PDF：页面分割与存储分层

导入时（CLI 或服务端）将 PDF 拆为三层，满足不同场景：

```
原始 PDF (100MB, 200页)
        │  导入时拆分
        ▼
┌─────────────────────────────────────────────┐
│  三层存储，全部存入 S3                         │
│                                              │
│  Layer 1: 缩略图 thumbs/page_001.jpg  10KB   │
│           → 书架列表 / 搜索网格                │
│                                              │
│  Layer 2: 预览图 images/page_001.webp  150KB │
│           → 快速翻页 / 搜索结果预览             │
│                                              │
│  Layer 3: 单页 PDF pages/page_001.pdf  300KB │
│           → Web 阅读器正文 (可选中文字)         │
│                                              │
│  Layer 0: 整本 original.pdf            50MB  │
│           → Flutter 客户端离线阅读 / 下载备份   │
└─────────────────────────────────────────────┘
```

| 层级 | 格式 | 大小/页 | 用途 | 使用端 |
|------|------|---------|------|--------|
| 缩略图 | JPG | ~10KB | 网格列表 | Web |
| 预览图 | WebP | ~150KB | 快速翻页 | Web |
| 单页 PDF | PDF | ~300KB | 可交互阅读 | Web |
| 整本 | PDF/EPUB | 原始大小 | 离线阅读 | Flutter |

**Web 阅读流程**：先显示 WebP 预览图（秒开），后台懒加载对应单页 PDF（加载完无缝替换）。用户翻页时瞬间展示图片，停下来阅读时文字可选中。

### S3 存储布局

```
s3://bucket-books/{document_id}/
├── original.{pdf,epub}
├── pages/
│   ├── 001.pdf        ← PDF：单页 PDF
│   └── ...
├── chapters/
│   ├── chapter-01.xhtml  ← EPUB：单章 HTML
│   └── ...
├── images/
│   ├── 001.webp
│   └── ...
└── thumbs/
    ├── 001.jpg
    └── ...
```

### EPUB：章节拆分与直接渲染

EPUB 本质是 ZIP 压缩的 HTML 合集。处理比 PDF 简单得多：

```
三体.epub (ZIP)
  → 解压
  → 解析 content.opf（元数据 + spine 阅读顺序）
  → 逐个提取 .xhtml 章节：
        chapter-01.xhtml → HTML → 去标签 → 纯文本 → jieba-rs 分词
        chapter-02.xhtml → HTML → 去标签 → 纯文本 → jieba-rs 分词
        ...
  → 存入：
        raw_text       = "第一章 科学边界 ..."    ← DB (搜索用)
        segmented_text = "第一章 科学 边界 ..."   ← DB (搜索用)
        chapter.xhtml  = 原始 HTML              ← S3 (Web 阅读用)
```

**EPUB 的优势（相比 PDF）：**

| | PDF | EPUB |
|---|---|---|
| 文字提取难度 | 中等（lopdf 解析） | **简单**（解压 + 去标签） |
| 是否一定有文字 | ⚠️ 可能是扫描件 | **一定有**（HTML 文本） |
| OCR 需要 | ⚠️ 扫描件需要 | **不需要** |
| "页"的概念 | 物理页 (page_num) | **章节** (page_num 映射到章节) |
| 服务端内存 | ~60MB | **~20MB** |
| Web 阅读 | PDF.js 渲染 | **浏览器原生渲染 HTML**（自动重排、字号缩放） |

**EPUB 在 Web 端的阅读体验天然优于 PDF**：浏览器直接渲染 XHTML 章节，文字可自动重排，字号随意放大缩小，不需要 PDF.js。

## 🌐 时间处理约定

| 层级 | 格式 | 说明 |
|------|------|------|
| **数据库** | `TIMESTAMPTZ` (UTC) | PG 统一存 UTC 时区时间戳 |
| **后端 API** | ISO 8601 UTC | `2026-07-13T10:30:00Z` |
| **前端展示** | 用户本地时区 | `new Date(utcString).toLocaleString()` 自动转换 |
| **同步冲突** | UTC 毫秒精度 | 跨设备对比 `modified` 时使用同一时区基准 |

```rust
// 后端写入：NOW() 已自动为 UTC（PG TIMESTAMPTZ 行为）
// 后端返回：序列化时保持 ISO 8601 UTC 格式
#[derive(Serialize)]
struct ProgressResponse {
    modified: DateTime<Utc>,  // chrono::DateTime<Utc> → "2026-07-13T10:30:00Z"
}
```

```javascript
// 前端展示
const utcTime = "2026-07-13T10:30:00Z";
const localStr = new Date(utcTime).toLocaleString('zh-CN');
// → "2026/7/13 18:30:00" （假设 UTC+8）
```

> **原则**：存储和传输统一 UTC，展示时转换为用户浏览器本地时区。不存储用户时区偏好（由浏览器 `Intl` API 自动处理）。

## 🔄 数据流设计

### 1. 文档上传流程

```
方式A：服务端直接处理（轻量 PDF）
─────────────────────────────────
用户上传 PDF
    ↓
服务端检测 PDF 类型
    ├─ 文字版 + CJK/空格语言 + <200MB → 服务端处理
    │   ├─ lopdf 提取文字
    │   ├─ jieba-rs 分词 / 空格切分
    │   ├─ 存 DB (raw_text + segmented_text)
    │   └─ 原文件 → S3
    │
    └─ 扫描件/大文件/特殊语言 → 返回提示
        "请使用 CLI 工具预处理: book-import ./文件.pdf --upload"

方式B：CLI 预处理导入（重型 PDF/EPUB/扫描件）
─────────────────────────────────────────
用户本地运行 CLI
    ↓
book-import ./书.epub
    ├─ OCR / 文字提取
    ├─ jieba-rs 分词
    ├─ 拆页渲染 (WebP + 单页 PDF + 缩略图)
    └─ 打包为 书.bookpkg
    ↓
上传 .bookpkg 到服务端
    ↓
服务端解包
    ├─ original.pdf/epub → S3
    ├─ pages/*.pdf → S3 (按页存取)
    ├─ images/*.webp → S3
    ├─ thumbs/*.jpg → S3
    └─ package.json → 逐页 INSERT 入库
    ↓
返回用户：导入成功
```

### 2. 用户标记流程

```
用户在阅读器中操作
    ↓
┌─────────────────┐
│ 选择操作类型           │
├────────────────────────┤
│ 高亮文本 → highlights表       │
│ 添加注释 → annotations表      │
│ 书签页面 → bookmarks表        │
└──────────────┘
    ↓
分词用户输入
    ↓
存储到PG
    ↓
异步索引到搜索引擎
    ↓
返回用户：操作成功
    ↓
源PDF文件保持不变（关键！）
```

### 3. 搜索流程（前端分词 + 服务端多层融合）

```
用户输入搜索词
    ↓
前端 jieba-rs WASM 分词
  "人工智能技术" → "人工 智能 技术"
    ↓  发送已分词查询到服务端
    ↓
┌──────────────────────────────────────┐
│ SEARCH_LAYER 环境变量决定融合策略       │
├──────────────────────────────────────┤
│                                      │
│ layer=like (默认)                     │
│   → 路径A: raw_text LIKE 单路        │
│                                      │
│ layer=tsvector                       │
│   → 路径B: tsvector GIN + 路径A 去重  │
│                                      │
│ layer=tantivy                        │
│   → 路径C: Tantivy BM25 打头         │
│   → 路径B: tsvector GIN 补充         │
│   → 路径A: LIKE 兜底                 │
│   → 三路并行 → 去重融合              │
│                                      │
│ layer=external                       │
│   → 外部引擎 + 路径A LIKE 双路融合     │
│                                      │
└──────────────┬───────────────────────┘
          返回: [{docId, pageNum, score, snippet}]
                  按相关性排序，高亮匹配词
```

## 💾 内存管理策略

### 启动时内存分布（512MB限制）

```
基础框架              ~8MB      ← Rust 显著优势
├─ Axum + Tower    3MB
├─ AWS SDK (Rust)  3MB
└─ 其他库       2MB

分词引擎          ~0-25MB    ← 懒加载，空闲时释放
├─ jieba-rs 词库     15MB   (仅处理PDF时加载)
├─ 自定义词库         5MB

数据访问层            ~15MB
├─ sqlx 连接池       5MB
├─ redis-rs 连接     3MB
└─ 序列化缓冲        7MB

运行时开销            ~25MB
├─ Tokio 运行时      6MB
├─ 堆分配           12MB
└─ 缓冲区            7MB

═══════════════════
空闲时总计           ~48MB（有 464MB 余量！）
处理PDF时总计        ~73MB（有 439MB 余量！）

开启 Tantivy (第 3 层): +20-50MB → 总计 ~100MB（仍有 412MB 余量）
```

* jieba-rs 懒加载：首次用时初始化，10 分钟无新任务自动释放。
* Tantivy 嵌入：同进程运行，非独立服务。通过 `SEARCH_LAYER=tantivy` 开启。

对比 Go 版 ~340MB，Rust 内存占用仅为 Go 的 **23%**
```

### 内存优势来源

| 优势 | 效果 | 原理 |
|------|------|------|
| **无 GC** | -60MB 堆 | 没有 GC 元数据开销，RAII 立即释放 |
| **枚举紧凑** | -15MB | `Option<Box<T>>` 编译为单指针 |
| **无 Goroutine 栈** | -30MB | Tokio task 初始 ~64 bytes（无栈协程） |
| **静态分发** | -10MB | 泛型单态化，无虚表 |

### 内存优化技巧（几乎不需要！）

| 优化项 | 效果 | 实现方式 |
|-----|------|---|
| **jieba-rs 精简词典** | -10MB | 仅加载高频词（可选） |
| **LRU缓存** | -5MB | 限制缓存条数 |
| **零拷贝解析** | 省 CPU | serde borrow 模式 |
