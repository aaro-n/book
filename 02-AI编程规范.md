# AI编程规范 - 确保代码质量和调试效率

## 核心原则

整个项目采用**AI友好的代码分割策略**，确保每个部分都是小的、独立的、易于调试的单元。

---

## 📏 代码规模限制

### 必须遵守的约束

| 指标 | 限制 | 原因 |
|------|------|------|
| 函数行数 | **<30行** | AI理解力强，逻辑清晰 |
| 结构体方法数 | **<5个** | 单一职责，易维护 |
| 文件行数 | **<200行** | 模块化，查看全貌 |
| 嵌套深度 | **<3层** | 逻辑清晰，减少复杂度 |
| 参数个数 | **<5个** | 如需多参数，用结构体 |
| 分支条件 | **<4个** | 如需复杂逻辑，分解函数 |

### ❌ 反面示例（避免）
```go
// 问题1：函数过长（70行）
func ProcessDocument(docID int, userID int, s3client *s3.Client, 
    db *sql.DB, cache *redis.Client, config Config) (map[string]interface{}, error) {
    // 混合了PDF处理、数据库操作、缓存、S3访问...
    // 50+ 行代码，AI难以理解和调试
}

// 问题2：参数过多
func SaveHighlight(docID int, userID int, pageNum int, color string, 
    text string, note string, position interface{}, ...) error {
    // 需要记住6个参数的含义和顺序
}

// 问题3：嵌套过深
if err == nil {
    if response.Code == 0 {
        if len(result) > 0 {
            if value, ok := result[0].(string); ok {
                // 4层嵌套，难以跟踪
            }
        }
    }
}
```

### ✅ 正确示例（推荐）
```go
// 方案1：单一职责，函数分解
func FetchDocumentFromS3(ctx context.Context, s3Key string) ([]byte, error) {
    // 只做S3操作，<20行
}

func ExtractTextFromPDF(pdfData []byte) ([]string, error) {
    // 只做PDF处理，<20行
}

func SaveDocumentContent(ctx context.Context, docID int, content []string) error {
    // 只做数据库操作，<20行
}

// 方案2：用结构体替代多参数
type HighlightInput struct {
    DocumentID   int
    UserID       int
    PageNum      int
    Color        string
    HighlightedText string
    Note     string
    Position     map[string]interface{}
}

func SaveHighlight(input *HighlightInput) error {
    // 参数清晰，易于理解
}

// 方案3：优先使用早期return避免嵌套
if err != nil {
    return err  // 早期return，减少嵌套
}

if response.Code != 0 {
    return errors.New("invalid response")
}

if len(result) == 0 {
    return errors.New("empty result")
}

// 主逻辑
value := result[0].(string)
```

### 每层的职责边界

**Models** （数据定义）
```go
// internal/models/document.go - 清晰的结构体定义
type Document struct {
    ID        int    `json:"id"`
    UserID    int       `json:"user_id"`
    Title     string    `json:"title"`
    S3Key     string    `json:"s3_key"`
    PageCount int       `json:"page_count"`
    CreatedAt time.Time `json:"created_at"`
}
```

**Repository** （数据访问）
```go
// internal/repository/document_repo.go - 纯数据库操作
func (r *DocumentRepository) GetByID(ctx context.Context, id int) (*models.Document, error) {
    // 仅做数据库查询，不涉及业务逻辑
}

func (r *DocumentRepository) Create(ctx context.Context, doc *models.Document) error {
    // 仅做插入操作
}
```

**Service** （业务逻辑）
```go
// internal/service/document_service.go - 业务逻辑编排
func (s *DocumentService) ProcessNewDocument(ctx context.Context, docID int) error {
    // 1. 从S3获取PDF
    doc, err := s.repo.GetByID(ctx, docID)
    if err != nil {
      return err
    }
    
    // 2. 提取文本
    pdfData, err := s.s3Service.GetObject(ctx, doc.S3Key)
    if err != nil {
        return err
    }
    
    // 3. 分词
    content, err := s.pdfService.ExtractText(pdfData)
    if err != nil {
        return err
    }
    
    // 4. 保存
    return s.repo.SaveContent(ctx, docID, content)
}
```

**Handler** （HTTP处理）
```go
// internal/handler/document_handler.go - 仅处理HTTP逻辑
func (h *DocumentHandler) Upload(c *gin.Context) {
    // 1. 解析请求
    file, err := c.FormFile("file")
    if err != nil {
        c.JSON(400, gin.H{"error": err.Error()})
        return
    }
    
    // 2. 调用Service
    docID, err := h.service.UploadDocument(c.Request.Context(), file)
    if err != nil {
        c.JSON(500, gin.H{"error": err.Error()})
        return
    }
    
    // 3. 返回响应
    c.JSON(200, gin.H{"document_id": docID})
}
```

## 🔄 错误处理规范

### 清晰的错误定义

```go
// internal/util/errors.go
package util

type ErrorCode string

const (
    ErrDocumentNotFound   ErrorCode = "DOC_NOT_FOUND"
    ErrUnauthorized       ErrorCode = "UNAUTHORIZED"
    ErrInvalidPDF         ErrorCode = "INVALID_PDF"
    ErrS3Upload          ErrorCode = "S3_UPLOAD_FAILED"
    ErrDatabaseError     ErrorCode = "DATABASE_ERROR"
)

type AppError struct {
    Code    ErrorCode
    Message string
    Details string
    Err     error  // 原始错误，便于调试
}

func (e *AppError) Error() string {
    return fmt.Sprintf("[%s] %s: %s", e.Code, e.Message, e.Details)
}

func NewError(code ErrorCode, message string) *AppError {
    return &AppError{
        Code:    code,
        Message: message,
    }
}
```

### 规范的错误处理

```go
// ✅ 好的做法
func GetDocument(ctx context.Context, id int) (*Document, error) {
    doc, err := repo.GetByID(ctx, id)
    if err != nil {
        return nil, NewError(ErrDocumentNotFound, "document not found").WithErr(err)
    }
    return doc, nil
}

// ❌ 避免
func GetDocument(ctx context.Context, id int) (*Document, error) {
    doc, err := repo.GetByID(ctx, id)
    if err != nil {
        // 丢失错误信息，难以调试
        return nil, fmt.Errorf("error")
    }
    return doc, nil
}
```

```markdown

- 输入参数及类型
- 返回值及错误情况
- 边界条件处理

## 位置
- 文件：internal/[module]/[file].go
- 相关模型：internal/models/[model].go
- 依赖服务：internal/service/[service].go

## 约束条件
- 代码行数：不超过30行
- 最大嵌套深度：2层
- 必须包含错误处理
- 需要简单的单元测试(<50行)

## 参考实现
- 相似功能：[文件名]
- 数据模型：[结构体定义]
- 错误类型：[ErrorCode定义]

## 特殊注意
- 这个模块会被XX模块调用
- 需要考虑XX的并发场景
- XX性能要求<100ms
```

## 📝 函数签名规范

### 推荐的函数设计

```go
// ✅ 函数命名清晰
func (r *DocumentRepository) GetByID(ctx context.Context, id int) (*Document, error)
func (s *DocumentService) ProcessDocument(ctx context.Context, docID int) error
func (h *DocumentHandler) GetDocument(c *gin.Context)

// ✅ 参数顺序固定
// 1. context
// 2. 业务参数
// 3. 可选配置（用结构体）

// ✅ 返回值顺序固定
// 1. 业务返回值
// 2. error

// ❌ 避免的做法
func Get(d int) (interface{}, interface{})  // 返回值不明确
func Process(a, b, c, d, e int) error      // 参数过多
func Handle()                      // 没有明确的输入输出
```

## 🔍 代码审查清单

在AI生成代码后，请检查以下项目：

- [ ] **行数限制** - 函数<30行，文件<200行
- [ ] **单一职责** - 每个函数做一件事
- [ ] **错误处理** - 所有error都被处理
- [ ] **命名清晰** - 变量和函数名易于理解
- [ ] **无重复代码** - 相似逻辑已提取
- [ ] **类型安全** - 避免interface{}
- [ ] **并发安全** - 如涉及并发，有适当的同步
- [ ] **性能考虑** - 没有明显的性能问题（如N+1查询）
- [ ] **注释完整** - Public函数有godoc注释
- [ ] **测试覆盖** - 核心逻辑有单元测试

## 🚀 AI编程工作流

### 标准流程

1. **需求定义**（您） → 列出功能点和约束
2. **数据模型**（AI） → 生成model.go（<100行）
3. **数据访问**（AI） → 生成repository.go（<120行）
4. **业务逻辑**（AI） → 生成service.go（<150行）
5. **HTTP处理**（AI） → 生成handler.go（<100行）
6. **单元测试**（AI） → 生成_test.go（<80行）
7. **集成测试**（您） → 手工验证端到端流程
8. **代码审查**（您） → 按照检查清单审查

### 逐步迭代

每完成一个小功能就进行：
- 本地测试验证
- 代码审查
- 提交到版本控制

避免一次性实现大功能，容易出错。

## 📚 命名约定

### 包名
```go
package models     // 简短，全小写，无下划线
package repository // 明确的功能
package util       // 通用工具
```

### 文件名
```go
user_repo.go       // repository_[domain].go
auth_service.go    // [domain]_service.go
document_handler.go // [domain]_handler.go
errors.go          // 通用工具文件
```

### 常量和变量
```go
// 常量：全大写，下划线分隔
const (
    ErrDocumentNotFound = "DOC_NOT_FOUND"
    MaxPageSize = 1000
)

// 变量：驼峰式，首字母表示可见性
var globalConfig *Config     // 包全局
ctx := context.Background()  // 局部变量
```

### 接口名
```go
// 接口以er结尾
type Reader interface { Read() error }
type Writer interface { Write() error }
type Segmenter interface { Segment(text string) []string }
```

## ✅ 总结

遵循这些规范的目的：
1. **AI易理解** - 代码简洁，逻辑清晰
2. **易调试** - 每个模块独立，问题易定位
3. **易维护** - 单一职责，改动范围小
4. **易扩展** - 模块化设计，新功能易添加
5. **易测试** - 函数纯净，无副作用

**记住：小的、专注的函数比大而全的函数更适合AI编程。**
